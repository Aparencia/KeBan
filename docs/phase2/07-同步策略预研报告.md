## 一、同步方案对比

### 1.1 候选方案总览

课伴需要同步的数据类型包括：笔记内容、闪卡数据、学习记录、牌组结构等。目标场景为**单人多设备同步**（MVP），后期可能扩展至**多人协同编辑**（V2.0）。

### 1.2 详细对比矩阵

| 维度 | CRDT (Yjs) | CRDT (Automerge) | Last-Write-Wins + 版本向量 | OT |
|------|-----------|-----------------|------------------------|-----|
| 数据一致性 | 强一致（数学证明） | 强一致 | 最终一致（冲突可能丢失） | 强一致 |
| 实现复杂度 | 高 | 中高 | 低 | 极高 |
| 学习曲线 | 2-3周 | 1-2周 | 3-5天 | 4-6周 |
| 性能（小文档 <10KB） | 优秀 | 良好 | 优秀 | 良好 |
| 性能（大文档 >100KB） | 需优化（CRDT膨胀） | 良好 | 优秀 | 良好 |
| 离线支持 | 原生支持 | 原生支持 | 需自建操作队列 | 不适合离线 |
| 冲突解决 | 自动（无冲突） | 自动 | 后写入覆盖 | 自动 |
| 库大小 | Yjs ~150KB | Automerge ~400KB | 自研 ~10KB | N/A |
| 社区活跃度 | 高（GitHub 15k+ star） | 中（GitHub 10k+） | N/A | 低 |
| React 集成 | y-react 可用 | @automerge/automerge-repo | 自行实现 | 需自研 |
| 后端依赖 | 可选（y-websocket） | 可选（sync server） | 需自建同步服务 | 必须中心服务器 |
| 适合场景 | 多人协同编辑 | 多人协同编辑 | 单人多设备同步 | 多人实时协同 |

### 1.3 各方案优劣分析

**CRDT (Yjs)**：
- 优势：数学上保证无冲突合并、原生离线支持、活跃的社区生态
- 劣势：CRDT 元数据膨胀（操作历史会持续增大，需定期压缩）、学习成本高、单人同步场景过于"重型"

**CRDT (Automerge)**：
- 优势：API 更友好、自动压缩历史、Rust 实现性能好
- 劣势：库体积大（~400KB WASM）、社区相比 Yjs 小、文档较少

**Last-Write-Wins + 版本向量**：
- 优势：实现简单直观、性能开销极小、无额外依赖、适合单人多设备
- 劣势：冲突时数据会丢失（后写覆盖前写）、不支持并发编辑合并

**OT (Operational Transformation)**：
- 优势：Google Docs 验证过的成熟方案
- 劣势：实现极其复杂、必须依赖中心服务器、不适合离线优先架构

## 二、模拟冲突场景测试（推断结果）

### 2.1 场景设计

| 场景 | 描述 | 并发度 | 数据影响 |
|------|------|--------|----------|
| 场景1 | 两设备同时编辑同一笔记的不同部分 | 低 | 笔记内容 |
| 场景2 | 两设备同时编辑同一笔记的相同部分 | 中 | 笔记内容 |
| 场景3 | 一设备离线新增100条闪卡，另一设备删除牌组 | 高 | 闪卡+牌组结构 |
| 场景4 | 弱网环境下批量操作（200ms延迟+5%丢包） | 中 | 全类型 |

### 2.2 场景1：不同部分编辑

> 设备A修改笔记标题，设备B修改笔记正文

| 方案 | 结果 | 数据完整性 | 用户感知 |
|------|------|-----------|----------|
| Yjs | ✅ 两部分修改均保留 | 完整 | 无缝合并 |
| Automerge | ✅ 两部分修改均保留 | 完整 | 无缝合并 |
| LWW + 版本向量 | ⚠️ 后写入的覆盖整个笔记 | **标题或正文丢失** | 需手动恢复 |
| OT | ✅ 两部分修改均保留 | 完整 | 无缝合并 |

### 2.3 场景2：相同部分编辑

> 设备A和B同时修改笔记的同一句话

| 方案 | 结果 | 数据完整性 | 用户感知 |
|------|------|-----------|----------|
| Yjs | ✅ 按字符级自动合并（可能产生不理想但无丢失的结果） | 完整 | 可能需要手动微调 |
| Automerge | ✅ 类似Yjs，按文本块合并 | 完整 | 可能需要手动微调 |
| LWW + 版本向量 | ⚠️ 后写入覆盖 | **一个修改丢失** | 需手动恢复 |
| OT | ✅ 服务端计算最优合并 | 完整 | 可能需要手动微调 |

### 2.4 场景3：离线批量操作 + 牌组删除

> 设备A离线新增100张闪卡到牌组X，设备B删除牌组X

| 方案 | 结果 | 数据完整性 | 用户感知 |
|------|------|-----------|----------|
| Yjs | ⚠️ 牌组恢复（删除被撤销），100张卡保留 | 完整但可能不符合预期 | 牌组意外恢复 |
| Automerge | ⚠️ 类似Yjs行为 | 完整但可能不符合预期 | 牌组意外恢复 |
| LWW + 版本向量 | ⚠️ 取决于操作时序：牌组删除在后→100张卡丢失；新增在后→牌组恢复 | **可能丢失数据** | 不可预测 |
| OT | 不适用（需在线） | — | — |

### 2.5 场景4：弱网环境批量操作

| 方案 | 操作成功率 | 最终一致性 | 平均同步延迟 |
|------|-----------|-----------|-------------|
| Yjs | 100%（本地优先） | ✅ 保证 | ~350ms（重连后） |
| Automerge | 100%（本地优先） | ✅ 保证 | ~400ms（重连后） |
| LWW + 版本向量 | 98.5%（重试后100%） | ⚠️ 大部分场景保证 | ~200ms |
| OT | 85%（需中心服务器） | ✅ 保证 | ~500ms |

### 2.6 场景测试结论

- **CRDT 方案**在文本级合并上表现优秀，但在实体级操作（删除牌组vs新增卡片）上行为不符合直觉
- **LWW** 在单人多设备场景下，冲突概率极低（同一用户很少在两个设备同时编辑同一内容），风险可接受
- 所有方案在弱网下都需要额外的重试与冲突检测机制

## 三、增量同步协议设计草案

### 3.1 数据模型

```typescript
// 操作日志格式
interface SyncOperation {
  id: string;              // 操作唯一ID (UUID)
  deviceId: string;        // 设备ID
  timestamp: number;       // Unix时间戳(ms)
  entityType: 'note' | 'card' | 'deck' | 'review_log' | 'setting';
  entityId: string;        // 被操作的实体ID
  operation: 'create' | 'update' | 'delete';
  payload?: any;           // 变更内容（增量diff或完整快照）
  vectorClock: Record<string, number>; // 版本向量 {deviceId: counter}
}

// 版本向量示例
interface VersionVector {
  [deviceId: string]: number; // 每个设备的操作计数器
}
```

### 3.2 同步握手协议

```
客户端A                        服务端                        客户端B
   │                            │                            │
   ├── POST /sync/handshake ───>│                            │
   │   {deviceId, vectorClock}  │                            │
   │                            │                            │
   │<── 200 OK ────────────────│                            │
   │   {serverVectorClock,      │                            │
   │    pendingOps: [...]}      │                            │
   │                            │                            │
   ├── POST /sync/push ────────>│                            │
   │   {ops: [本地新操作...]}   │                            │
   │                            │                            │
   │<── 200 OK ────────────────│                            │
   │   {conflicts: [],          │────── 推送变更 ────────────>│
   │    ackedOps: [...]}        │                            │
   │                            │                            │
```

### 3.3 增量推送策略

| 策略 | 触发条件 | 推送内容 | 优先级 |
|------|----------|----------|--------|
| 实时推送 | 在线状态下的每次操作 | 单条操作 | 高 |
| 批量推送 | 离线恢复后 / 弱网检测 | 累积的所有未同步操作 | 中 |
| 定时轮询 | 每5分钟（兜底） | 检查是否有遗漏的变更 | 低 |
| 手动同步 | 用户主动触发 | 全量差异比对 | 低 |

### 3.4 冲突检测与处理（LWW 方案）

```typescript
function resolveConflict(
  localOp: SyncOperation, 
  remoteOp: SyncOperation
): SyncOperation {
  // LWW策略：时间戳较大的操作胜出
  if (localOp.timestamp > remoteOp.timestamp) {
    return localOp; // 本地操作胜出
  } else {
    return remoteOp; // 远程操作胜出
  }
}

// 增强版：实体级 LWW + 字段级合并
function smartMerge(
  localEntity: Entity, 
  remoteEntity: Entity,
  localClock: VersionVector,
  remoteClock: VersionVector
): Entity {
  const merged = { ...localEntity };
  
  for (const field of Object.keys(remoteEntity)) {
    // 如果远程版本在该字段上更新，且本地未修改该字段
    if (remoteEntity._fieldClock?.[field] > (localEntity._fieldClock?.[field] || 0)) {
      merged[field] = remoteEntity[field];
    }
  }
  
  return merged;
}
```

### 3.5 同步状态管理

```typescript
interface SyncState {
  status: 'idle' | 'syncing' | 'conflict' | 'offline' | 'error';
  lastSyncTime: number;
  pendingOpsCount: number;
  vectorClock: VersionVector;
  conflictLog: ConflictRecord[];  // 冲突日志，供用户手动解决
}
```

## 四、选型结论

### 4.1 MVP-2 采用 LWW + 版本向量

| 决策因素 | 评估 | 说明 |
|----------|------|------|
| 目标场景匹配 | ⭐⭐⭐⭐⭐ | MVP阶段为单人多设备同步，非多人协同 |
| 实现成本 | ⭐⭐⭐⭐⭐ | 3-5天学习+1-2周实现，自研代码量<500行 |
| 性能表现 | ⭐⭐⭐⭐⭐ | 无CRDT元数据膨胀，同步延迟最低 |
| 冲突风险 | ⭐⭐⭐⭐ | 单人场景冲突概率极低（<1%） |
| 可维护性 | ⭐⭐⭐⭐ | 代码逻辑简单，易于调试和维护 |
| 包大小影响 | ⭐⭐⭐⭐⭐ | 自研~10KB，相比Yjs/ Automerge 极小 |

**核心理由**：

1. **场景匹配**：课伴 MVP 阶段是单人多设备同步，不存在"两人同时编辑同一笔记"的场景，CRDT 的核心优势（无冲突合并）在此场景下价值有限
2. **成本可控**：LWW 实现简单，可在2周内完成，不阻塞核心功能开发
3. **性能最优**：无 CRDT 元数据开销，同步速度最快
4. **风险可接受**：单人场景下冲突概率极低，且关键数据（学习记录）天然无冲突

### 4.2 冲突风险缓解措施

1. **操作粒度细化**：以字段级而非实体级做 LWW，降低数据丢失概率
2. **冲突日志记录**：所有被覆盖的操作记录到日志，支持用户手动恢复
3. **软删除机制**：删除操作改为标记删除+延迟物理删除（7天），防止误覆盖
4. **离线队列**：离线操作排队，上线后按时间戳顺序应用

### 4.3 V2.0 升级路径：引入 Yjs

当课伴发展到多人协同场景（如共享牌组、协同编辑笔记）时：

```
MVP-2 (LWW)                    V2.0 (LWW + Yjs 混合)
┌──────────────┐               ┌──────────────────────┐
│ 全部数据      │               │ 结构化数据（闪卡/牌组）│
│ LWW同步       │               │ → 仍用LWW（天然无冲突）│
└──────────────┘               ├──────────────────────┤
                               │ 笔记富文本内容        │
                               │ → 引入Yjs协同编辑     │
                               ├──────────────────────┤
                               │ 学习记录              │
                               │ → 仍用LWW（追加型数据）│
                               └──────────────────────┘
```

**迁移策略**：
1. 保持 LWW 同步框架不变，新增 Yjs 协同编辑模块
2. 仅在需要协同的笔记内容上使用 Yjs
3. 结构化数据（闪卡、牌组、学习记录）继续使用 LWW
4. 渐进式引入，不影响已有功能
