> 文档版本：v2.0  
> 创建日期：2026-07-08  
> 所属阶段：阶段二（签约与项目计划）  
> 风险评估方法：概率-影响矩阵（P×I），风险值 = 概率(1-5) × 影响(1-5)

## 一、风险评级标准

### 概率等级

| 等级 | 描述 | 说明 |
|------|------|------|
| 5 | 几乎确定 | > 90% 可能发生 |
| 4 | 很可能 | 60%–90% |
| 3 | 可能 | 30%–60% |
| 2 | 不太可能 | 10%–30% |
| 1 | 罕见 | < 10% |

### 影响等级

| 等级 | 描述 | 说明 |
|------|------|------|
| 5 | 灾难性 | 项目无法继续或核心功能完全不可用 |
| 4 | 严重 | 核心功能严重受损，工期延长 > 8周 |
| 3 | 中等 | 部分功能受影响，工期延长 3-8周 |
| 2 | 轻微 | 功能轻微受损，工期延长 < 3周 |
| 1 | 可忽略 | 几乎无影响 |

### 风险值等级

| 风险值 | 等级 | 响应要求 |
|--------|------|----------|
| 15–25 | **极高** | 立即行动，专项应对方案 |
| 10–14 | **高** | 本周期内制定应对方案 |
| 5–9 | **中** | 持续监控，预备应对方案 |
| 1–4 | **低** | 常规监控 |

### NR-1：本地存储方案无法满足性能需求

| 维度 | 详情 |
|------|------|
| **风险描述** | IndexedDB 或 SQLite WASM 方案在大数据量（10万+笔记/闪卡）场景下出现性能瓶颈，导致应用卡顿或数据丢失 |
| **影响** | 4（严重） |
| **概率** | 3（可能） |
| **风险值** | **12（高）** |
| **风险类别** | 技术风险 |
| **影响范围** | 全部本地功能（四模块+基础架构） |
| **应对策略** | **缓解**：M1.5 门禁前完成四方案基准测试；**转移**：混合方案（热数据 IndexedDB + 冷数据 SQLite WASM）作为后备；**应急**：降级为简化数据模型（限制本地数据量） |
| **责任人** | 前端架构师 |
| **触发条件** | M1.5 评审时所有方案均未通过性能基准 |
| **监控指标** | 存储操作 P99 延迟、内存占用峰值 |

### NR-2：同步冲突导致数据丢失

| 维度 | 详情 |
|------|------|
| **风险描述** | 多设备同时编辑同一数据时，冲突解决策略不当导致用户数据被覆盖或丢失，造成不可挽回的用户损失 |
| **影响** | 5（灾难性） |
| **概率** | 4（很可能） |
| **风险值** | **20（极高）** |
| **风险类别** | 技术风险 |
| **影响范围** | 云同步功能、用户数据安全、产品信誉 |
| **应对策略** | **缓解**：采用 CRDT（Yjs/Automerge）或版本向量 + LWW 方案，确保数据一致性 > 99.9%；**防御**：append-only 操作日志，所有变更可追溯、可回滚；**保底**：冲突时保留双方版本，由用户选择最终版本，永不自动删除；**验证**：M2 阶段完成 500 次并发编辑模拟测试 |
| **责任人** | 全栈开发工程师 |
| **触发条件** | 同步后数据一致性测试通过率 < 99.9% |
| **监控指标** | 冲突率、数据丢失事件数、同步成功率 |

**详细应对方案**：

```
冲突解决三层防护：
├── 第一层：预防 — 字段级锁粒度，最小化冲突面
├── 第二层：自动解决 — CRDT自动合并无冲突字段
└── 第三层：用户介入 — 冲突字段保留双方版本，用户选择
    └── 操作日志：append-only，支持任意时间点回溯
```

### NR-3：双模式代码维护成本翻倍

| 维度 | 详情 |
|------|------|
| **风险描述** | 在线模式与离线模式（双模式）的代码逻辑分裂导致维护成本翻倍，Bug修复需两套验证，新功能开发周期延长 50%+ |
| **影响** | 4（严重） |
| **概率** | 4（很可能） |
| **风险值** | **16（极高）** |
| **风险类别** | 架构风险 |
| **影响范围** | 全项目代码库、开发效率、长期维护成本 |
| **应对策略** | **缓解**：采用策略模式抽象，统一数据访问接口（Repository 模式），业务层代码不感知当前模式；**规范**：所有新功能必须同时通过在线/离线双模式测试方可合并；**工具**：双模式自动化测试框架，CI 中并行运行两种模式测试 |
| **责任人** | 前端架构师 + 技术负责人 |
| **触发条件** | 双模式相关 Bug 占比 > 30% 或新功能开发周期延长 > 50% |
| **监控指标** | 双模式 Bug 占比、模式切换相关崩溃率、代码重复率 |

**架构策略**：

```
统一数据访问层设计：
├── Repository接口层 — 业务代码只依赖接口，不感知实现
├── LocalRepository  — 本地模式实现（IndexedDB/SQLite）
├── CloudRepository  — 云端模式实现（API + 本地缓存）
└── HybridRepository — 双模式切换时自动路由（在线→Cloud，离线→Local）
```

| **责任人** | 前端开发 |
| **触发条件** | 与 Anki 交叉验证一致性 < 100% |
| **监控指标** | 用户平均记忆保持率、复习准时率 |
