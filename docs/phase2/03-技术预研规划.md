## 二、课题1：本地存储方案选型（预计3周）

### 2.1 预研目标

确定课伴本地优先架构的数据存储基座，需在以下候选方案中做出最终选型：
- **方案A**：IndexedDB（原生浏览器API / Dexie.js封装）
- **方案B**：SQLite WASM（sql.js / wa-sqlite）
- **方案C**：混合方案（热数据IndexedDB + 冷数据/复杂查询SQLite WASM）

### 2.2 验证项

#### 验证项1：IndexedDB 容量与性能实测

| 测试项 | 目标指标 | 测试方法 |
|--------|----------|----------|
| 单库容量上限 | ≥ 500MB（含附件） | 批量写入10万条笔记+10万条闪卡记录 |
| 写入性能（批量） | ≥ 1000条/秒 | 批量事务写入，记录耗时 |
| 查询性能（索引） | 单条查询 < 10ms | 按ID/标签/时间范围查询 |
| 全文搜索性能 | 10万条数据搜索 < 500ms | 模拟中英文混合搜索 |
| 内存占用 | 峰值 < 200MB | Chrome DevTools Memory面板监控 |

**实测浏览器**：Chrome 120+、Firefox 120+、Edge 120+、Safari 17+

**成功标准**：IndexedDB 可支撑 10万+笔记 + 10万+闪卡，核心查询延迟满足目标指标

#### 验证项2：SQLite WASM 方案对比

| 对比维度 | sql.js | wa-sqlite | 评估标准 |
|----------|--------|-----------|----------|
| WASM体积 | ~1MB | ~800KB | 越小越好 |
| 查询性能 | 基准 | 对比基准 | 优于IndexedDB 3倍 |
| 事务支持 | 完整 | 完整 | ACID兼容 |
| 数据持久化 | 需手动导出 | OPFS原生支持 | 自动持久化优先 |
| 浏览器兼容 | 全平台 | 需OPFS支持 | Chrome 102+/Edge 102+/Firefox 111+ |
| 社区活跃度 | GitHub Stars/Issues | GitHub Stars/Issues | 长期维护保障 |

**成功标准**：至少一个 SQLite WASM 方案在查询性能上优于 IndexedDB 3倍，且浏览器兼容性满足目标

#### 验证项3：浏览器兼容性矩阵

| 浏览器 | 版本要求 | IndexedDB | OPFS | SQLite WASM | Service Worker |
|--------|----------|-----------|------|-------------|----------------|
| Chrome | 110+ | ✅ | ✅ | ✅ | ✅ |
| Firefox | 115+ | ✅ | ⚠️(121+) | ✅ | ✅ |
| Edge | 110+ | ✅ | ✅ | ✅ | ✅ |
| Safari | 16.4+ | ✅ | ❌ | ⚠️ | ✅ |

**成功标准**：主方案需覆盖 ≥ 90% 目标用户浏览器（基于caniuse.com数据）

#### 验证项4：Service Worker 离线缓存

| 测试场景 | 预期结果 |
|----------|----------|
| 完全断网启动 | 应用完整加载，所有UI可用 |
| 断网状态下CRUD操作 | 数据写入本地队列，无报错 |
| 网络恢复后自动同步 | 离线操作自动重放，数据一致 |
| 缓存更新策略 | 新版本发布后 ≤ 24小时用户获取最新代码 |

**成功标准**：断网环境下应用完整可用，CRUD操作零失败

### 2.3 最终交付物

- [ ] 本地存储方案对比评估报告（含性能基准数据）
- [ ] 浏览器兼容性实测结果表

## 三、课题2：同步引擎方案验证（预计3周）

### 3.1 预研目标

验证多设备数据同步的核心可行性，确定冲突解决策略与同步协议方案。

### 3.2 验证项

#### 验证项1：CRDT vs Last-Write-Wins（LWW）对比

| 对比维度 | CRDT (Automerge/Yjs) | LWW + 版本向量 |
|----------|---------------------|---------------|
| 冲突解决 | 自动合并，无冲突 | 保留双方版本，手动/自动选胜 |
| 实现复杂度 | 高（需理解CRDT原理） | 中（时间戳+版本号） |
| 数据膨胀 | 中（元数据开销） | 低（仅版本号） |
| 离线支持 | 原生优秀 | 需额外实现操作队列 |
| 库成熟度 | Yjs/Automerge活跃 | 需自建 |
| 实时协作 | 天然支持 | 不支持 |

**测试方法**：
1. 模拟3台设备并发编辑同一笔记（500次操作序列）
2. 模拟1000张闪卡的批量学习记录同步
3. 断网→重连场景下数据一致性验证

**成功标准**：数据一致性 > 99.9%，无数据丢失

#### 验证项2：增量同步协议

| 测试场景 | 性能指标 | 验证方法 |
|----------|----------|----------|
| 弱网同步（3G模拟） | 同步延迟 < 10s | Chrome网络限速，同步100条变更 |
| 正常网络同步 | 同步延迟 < 5s | WiFi环境，同步1000条变更 |
| 断网重连同步 | 全量追平 < 15s | 断网5分钟后重连，验证增量同步 |
| 增量传输效率 | 仅传输diff数据 | 对比全量vs增量传输数据量 |

**成功标准**：正常网络下同步延迟 < 5s，弱网/断网/重连场景功能完整可用

#### 验证项3：离线操作队列

| 测试项 | 成功标准 |
|--------|----------|
| 离线操作积累 | 支持 ≥ 1000条离线操作入队 |
| 批量重放 | 1000条操作重放耗时 < 10s |
| 重放一致性 | 重放后数据与在线操作结果100%一致 |
| 队列持久化 | 浏览器重启后队列不丢失 |
| 冲突重放 | 冲突操作自动检测，保留双方版本 |

**成功标准**：1000条离线操作批量重放无数据丢失，耗时 < 10s

### 3.3 最终交付物

- [ ] CRDT vs LWW 对比评估报告（含实测数据）
- [ ] 同步引擎可演示原型（支持双设备同步）
- [ ] 增量同步协议设计文档
- [ ] 离线操作队列实现方案

## 四、课题3：SM-2算法与编辑器选型（预计2周）

### 4.1 预研目标

1. 验证 SM-2 间隔重复算法实现的正确性
2. 完成富文本编辑器的技术选型

### 4.2 验证项

#### 验证项1：SM-2 算法正确性验证

**基准实现**：Anki 开源 SM-2 实现（Python）

**验证方法**：
1. 用 TypeScript 实现完整 SM-2 算法（含EF调整、Interval计算、Repetitions管理）
2. 构建100组测试用例，覆盖所有边界场景：
   - 全新卡片首次学习
   - 连续正确回答（Easy/Good）
   - 错误回答（Again）后恢复
   - EF值极端场景（最低1.3、最高3.5）
   - 超长间隔（> 1年）后的复习
3. 与 Anki 开源实现交叉对比，逐条验证调度结果

**成功标准**：100组测试用例调度结果与 Anki 实现 **100% 一致**

**关键验证指标**：

| 指标 | 说明 | 目标 |
|------|------|------|
| EF（Easiness Factor）计算 | 难度因子调整公式 | 精确到小数点后4位 |
| Interval（间隔天数） | 复习间隔计算 | 整数天，与Anki一致 |
| Repetitions（重复次数） | 连续正确计数 | 与Anki完全一致 |
| 到期日计算 | 下次复习日期 | 日期精确匹配 |

#### 验证项2：富文本编辑器选型

| 候选方案 | ProseMirror | TipTap (基于ProseMirror) | Slate |
|----------|-------------|--------------------------|-------|
| **架构** | 底层框架 | 高层封装 | 独立框架 |
| **学习曲线** | 陡峭 | 平缓 | 中等 |
| **协作支持** | Yjs原生 | Yjs支持 | 社区方案 |
| **Markdown支持** | 需扩展 | 内置扩展 | 需扩展 |
| **模板系统** | 可定制 | 扩展友好 | 可定制 |
| **TypeScript** | 完整支持 | 完整支持 | 完整支持 |
| **社区生态** | 成熟 | 快速增长 | 中等 |
| **移动端适配** | 需自行处理 | 较好 | 需自行处理 |
| **包体积** | 中等（~150KB） | 较大（~300KB） | 中等（~200KB） |

**评估维度权重**：

| 维度 | 权重 | 说明 |
|------|------|------|
| 功能完备性 | 25% | 富文本、表格、代码块、数学公式 |
| 可扩展性 | 20% | 模板系统集成、自定义节点 |
| 协作能力 | 15% | 未来多人协作可能性 |
| 性能表现 | 15% | 大文档渲染、输入响应 |
| 开发效率 | 15% | API友好度、文档质量 |
| 社区与维护 | 10% | 长期维护保障 |

**成功标准**：完成加权评分矩阵，选出综合得分最高方案，并通过原型验证核心场景（模板渲染、大文档编辑、Markdown快捷输入）

### 4.3 最终交付物

## 五、课题4：AI接口层架构（预计1周）

### 5.1 预研目标

设计可插拔的 AI 服务架构，确保 AI 功能为增强项而非依赖项，实现优雅降级。

### 5.2 验证项

#### 验证项1：可插拔 AI 模块设计

**架构设计原则**：
1. **接口隔离**：所有 AI 功能通过统一接口层调用，业务层不感知具体 AI 提供商
2. **策略模式**：AI Provider 可热插拔切换（OpenAI / 文心一言 / 通义千问 / 本地模型）
3. **降级策略**：AI 服务不可用时，自动降级为本地基础功能，用户体验不中断
4. **成本控制**：请求限流、缓存、Token 预算管理

**接口定义草案**：

```typescript
interface AIService {
  // 笔记增强
  summarize(content: string): Promise<string>
  continueWriting(context: string): Promise<string>
  extractKeyPoints(content: string): Promise<string[]>

  // 闪卡增强
  generateCards(content: string): Promise<FlashCard[]>
  adjustDifficulty(history: ReviewHistory[]): Promise<DifficultyAdjust>

  // 费曼增强
  evaluateExplanation(original: string, explanation: string): Promise<Evaluation>

  // 番茄钟增强
  recommendSession(history: PomodoroRecord[]): Promise<SessionConfig>
}

interface AIProvider {
  name: string
  chat(messages: Message[]): Promise<string>
  isAvailable(): Promise<boolean>
  getCostEstimate(tokens: number): number
}
```

**成功标准**：接口定义完整，支持至少2个 AI Provider 实现切换

#### 验证项2：AI 服务响应时间

| AI 功能 | 目标延迟 | 测试方法 |
|---------|----------|----------|
| 笔记摘要 | < 3s | 1000字笔记输入，记录端到端延迟 |
| 智能续写 | < 3s | 500字上下文输入，记录首Token延迟+总延迟 |
| 卡片生成 | < 5s | 1000字教材内容，生成10张卡片 |
| 费曼评估 | < 3s | 原文+解释对比评估 |
| 番茄推荐 | < 1s | 30天历史数据输入 |

**成功标准**：所有 AI 功能端到端延迟 < 3秒（卡片生成 < 5秒）

#### 验证项3：降级策略原型

| 场景 | 降级行为 | 用户提示 |
|------|----------|----------|
| AI 服务超时（> 5s） | 取消请求，返回缓存结果或空 | "AI服务响应较慢，已使用离线结果" |
| AI 服务不可用 | 隐藏 AI 功能入口，保留基础功能 | "AI增强功能暂时不可用" |
| Token 配额耗尽 | 停止 AI 调用，提示用户 | "本月AI额度已用完，下月恢复" |
| 网络断开 | 自动切换本地模式 | "已切换离线模式，AI功能需联网" |

**成功标准**：所有降级场景均有明确策略，降级切换对用户无感知

### 5.3 最终交付物

- [ ] AI 接口层架构设计文档
- [ ] Provider 策略模式实现原型
- [ ] 降级策略完整定义与原型
- [ ] AI 服务响应时间实测报告
