# 课伴本地存储技术方案对比报告

> **项目**：课伴 KeBan  
> **阶段**：Phase 2 - 技术预研  
> **预研时间**：W07-W09（第7-9周）  
> **负责人**：前端技术组  
> **版本**：v1.0

---

## 一、方案概述

课伴作为浏览器端 PWA 应用，本地存储是核心基础设施。需要承载笔记、闪卡、学习记录等多种结构化数据，对容量、性能、查询能力均有较高要求。本预研对比以下四种方案：

| 方案 | 简介 |
|------|------|
| **IndexedDB (Dexie.js)** | 浏览器原生 NoSQL 数据库，Dexie.js 为其提供友好封装层 |
| **SQLite WASM (wa-sqlite)** | 将 SQLite 编译为 WebAssembly，通过 OPFS 持久化，提供完整 SQL 能力 |
| **OPFS** | Origin Private File System，浏览器提供的底层文件系统 API，适合大文件存储 |
| **localStorage** | 浏览器原生 key-value 存储，容量有限，仅适合配置项等小数据 |

## 二、详细对比评测

### 2.1 核心维度对比

| 维度 | IndexedDB (Dexie.js) | SQLite WASM (wa-sqlite) | OPFS | localStorage |
|------|---------------------|------------------------|------|-------------|
| 浏览器支持 | Chrome 12+/FF 16+/Safari 10+/Edge 12+ | Chrome 109+/FF 113+/Safari 16.4+ | Chrome 102+/Edge 102+（无Safari/FF） | 全平台 |
| 存储容量 | 50MB-2GB（浏览器相关） | 受OPFS限制（可达GB级） | 理论无限制 | 5-10MB |
| 读取性能 | ~0.5ms/条 | ~0.1ms/条 | N/A（文件系统） | ~0.01ms/条 |
| 写入性能 | ~1ms/条 | ~0.3ms/条 | 顺序写入极快 | ~0.1ms/条 |
| 查询能力 | 索引查询、范围查询 | 完整SQL | 无（需自行索引） | 仅key-value |
| 事务支持 | 完整事务 | 完整ACID事务 | 无 | 无 |
| 额外加载 | 0（Dexie.js ~15KB） | ~1MB WASM二进制 | 0 | 0 |
| 成熟度 | 高（2010年标准化） | 中（wa-sqlite 2022年发布） | 低（2022年实验性） | 高 |
| API 易用性 | 高（Dexie封装后接近ORM） | 中（需SQL知识） | 低（文件读写API） | 高 |
| 数据迁移 | 简单（JSON导入导出） | 中等（数据库文件迁移） | 困难 | 简单 |

### 2.2 开发体验评估

**IndexedDB (Dexie.js)**：
```typescript
// Dexie.js 使用示例
const db = new Dexie('keban');
db.version(1).stores({
  notes: '++id, title, deckId, updatedAt',
  cards: '++id, deckId, dueDate, [deckId+dueDate]'
});

// 查询到期卡片
const dueCards = await db.cards
  .where('dueDate')
  .below(new Date())
  .toArray();
```

**SQLite WASM**：
```typescript
// wa-sqlite 使用示例
const db = await waSqlite.open('keban.db');
await db.exec(`
  SELECT * FROM cards 
  WHERE dueDate <= date('now')
  ORDER BY dueDate ASC
`);
```

## 三、容量实测数据（基于行业公开基准）

### 3.1 各浏览器存储容量上限

| 浏览器 | IndexedDB 上限 | OPFS 上限 | localStorage 上限 | 备注 |
|--------|---------------|-----------|------------------|------|
| Chrome 120+ | 磁盘空间的 ~10%（通常 5-50GB） | 与IndexedDB共享配额 | 5MB | 超出后触发 QuotaExceededError |
| Firefox 121+ | ~2GB（可申请扩容） | 不支持 | 5MB | Firefox 会在配额紧张时主动清理 |
| Safari 17+ | ~1GB | 不支持 | 5MB | **7天不活跃可能清除全部数据** |
| Edge 120+ | 与Chrome类似 | 与Chrome类似 | 5MB | 基于Chromium，行为一致 |

### 3.2 课伴数据量估算

| 数据类型 | 单条大小（估算） | 预估用户数据量 | 总占用 |
|----------|-----------------|---------------|--------|
| 笔记（富文本） | 2-10KB | 500-5000条 | 1-50MB |
| 闪卡（含模板数据） | 0.5-2KB | 1000-50000条 | 0.5-100MB |
| 学习记录 | 0.1-0.3KB | 10000-500000条 | 1-150MB |
| 媒体资源（图片/音频） | 50KB-2MB | 100-5000个 | 5MB-10GB |
| **合计** | — | — | **约 12MB-10.3GB** |

**结论**：对于重度用户（50000+闪卡+大量媒体），IndexedDB 容量可能紧张，需要配合 OPFS 存储大文件。

### 3.3 容量告警与监控策略

| 使用率 | 状态 | 处理策略 |
|--------|------|----------|
| < 50% | 正常 | 无需干预 |
| 50%-80% | 警告 | 提醒用户清理媒体缓存，压缩旧笔记 |
| 80%-95% | 危险 | 自动清理过期学习记录缓存，提示用户备份 |
| > 95% | 紧急 | 强制备份到云端，停止写入新数据 |

通过 `navigator.storage.estimate()` 可获取当前配额和使用量，实现实时监控。

## 四、性能基准测试（模拟数据，基于行业基准）

### 4.1 十万条笔记 CRUD 性能

测试数据规模：100,000 条笔记，每条平均 5KB 富文本内容。

| 操作 | IndexedDB (Dexie.js) | SQLite WASM | localStorage |
|------|---------------------|-------------|-------------|
| 批量写入（10万条） | ~95s（逐条）/ ~12s（批量事务） | ~28s（事务内批量INSERT） | 不可行（超出容量） |
| 单条读取（按ID） | ~0.4ms | ~0.08ms | ~0.01ms |
| 单条写入 | ~0.8ms | ~0.25ms | ~0.1ms |
| 全表扫描 | ~3200ms | ~450ms | 不可行 |
| 索引查询（按deckId） | ~2ms | ~0.5ms | 不可行 |
| 范围查询（按日期） | ~5ms | ~1ms | 不可行 |
| 复合条件查询 | ~15ms | ~2ms | 不可行 |

### 4.2 十万条闪卡查询性能

测试场景：从 100,000 张闪卡中查询到期卡片（假设 5000 张到期）。

| 查询场景 | IndexedDB (Dexie.js) | SQLite WASM |
|----------|---------------------|-------------|
| 查询到期卡片 | ~8ms | ~1.5ms |
| 按牌组+到期日复合查询 | ~12ms | ~2ms |
| 学习统计聚合（COUNT/SUM） | ~45ms（需应用层聚合） | ~3ms（SQL原生聚合） |
| 全文搜索（标题） | ~200ms（需自建索引） | ~50ms（FTS5） |

### 4.3 批量写入性能（事务内）

| 批量大小 | IndexedDB (Dexie.js) | SQLite WASM |
|----------|---------------------|-------------|
| 100条 | ~15ms | ~5ms |
| 1,000条 | ~120ms | ~35ms |
| 10,000条 | ~1.1s | ~0.3s |
| 100,000条 | ~12s | ~3.5s |

### 4.4 性能评估小结

- SQLite WASM 在查询性能上优势明显（3-10x），特别是复杂查询和聚合
- IndexedDB 在简单读写场景下性能足够，批量事务写入也能接受
- 对于课伴的数据规模，IndexedDB 性能瓶颈不会成为主要问题

### 5.3 关键风险点

1. **Safari OPFS 不支持**：SQLite WASM 方案在 Safari 上无法使用 OPFS 持久化，直接排除为通用方案
2. **Safari 7天清除策略**：Safari 会在用户7天不访问后清除该站点的所有本地数据（包括IndexedDB、Cache API、Service Worker等），对PWA应用影响重大
3. **Safari Blob 存储限制**：Safari 中 IndexedDB 存储 Blob 对象有额外大小限制
4. **Firefox OPFS 限制**：Firefox 113+ 支持 OPFS 但存在稳定性问题，不建议生产使用
5. **iOS WebView 限制**：iOS 上所有 WebView（包括 PWA 安装后）共享 Safari 引擎，继承所有 Safari 限制
6. **隐身模式差异**：部分浏览器隐身模式下 IndexedDB 配额极低或完全不可用

## 六、选型结论

### 6.1 推荐方案：IndexedDB (Dexie.js) 作为主存储

**选型理由**：

| 决策因素 | 评分 | 说明 |
|----------|------|------|
| 浏览器兼容性 | ⭐⭐⭐⭐⭐ | 唯一在所有目标浏览器中稳定工作的方案 |
| 容量充足性 | ⭐⭐⭐⭐ | 满足大多数用户，重度用户需配合 OPFS 存媒体 |
| API 开发效率 | ⭐⭐⭐⭐⭐ | Dexie.js 封装后开发体验优秀，接近 ORM |
| 查询能力 | ⭐⭐⭐ | 索引+范围查询覆盖基本需求，复杂聚合需应用层处理 |
| 生态成熟度 | ⭐⭐⭐⭐⭐ | 标准化早、社区庞大、生产验证充分 |
| 离线可靠性 | ⭐⭐⭐⭐⭐ | 无 Safari 清除风险 |

### 6.2 Safari 风险缓解策略

1. **Storage API 持久化申请**：调用 `navigator.storage.persist()` 降低被清除概率
2. **活跃检测与提醒**：检测用户超过5天未访问时推送通知提醒
3. **云端备份兜底**：核心数据（学习进度、牌组结构）同步到云端，即使本地被清除也可恢复
4. **Service Worker 保活**：周期性后台同步保持站点活跃度（需评估 Apple 政策）

### 6.3 混合存储策略

```
┌─────────────────────────────────────┐
│           Dexie.js (IndexedDB)       │
│  • 笔记元数据与富文本内容             │
│  • 闪卡数据与模板定义                 │
│  • 学习记录与统计数据                 │
│  • 用户配置与偏好设置                 │
├─────────────────────────────────────┤
│           OPFS (可选增强)            │
│  • 大尺寸媒体文件（图片、音频）       │
│  • 导出备份文件暂存                   │
├─────────────────────────────────────┤
│           localStorage               │
│  • 主题/语言等轻量配置                │
│  • 首次使用标记                       │
│  • 同步状态标记                       │
└─────────────────────────────────────┘
```

### 6.4 后期迁移路径（V2.0 考虑）

当 wa-sqlite 和 OPFS 在 Safari 上稳定后，可考虑迁移到 SQLite WASM：

1. **数据适配层抽象**：在业务代码和存储层之间建立 Repository 接口
2. **双写验证**：迁移期间同时写 IndexedDB 和 SQLite，验证数据一致性
3. **渐进切换**：先在新用户上启用 SQLite，验证稳定后逐步迁移老用户
4. **回退机制**：保留 IndexedDB 作为降级方案
