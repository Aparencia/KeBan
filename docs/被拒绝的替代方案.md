# 课伴项目被拒绝的替代方案

本文档记录课伴项目规划阶段评估后明确拒绝的 8 项替代方案，包括每项方案的描述、考虑场景、拒绝理由及未来重新评估的条件。记录这些决策有助于后续团队成员理解"为什么不选 X"，避免重复讨论。

---

## 1. 云端优先架构（原方案）

### 方案描述

云端优先架构是指将数据存储、用户认证、核心业务逻辑全部放在服务端，客户端（浏览器/App）仅作为展示层。用户必须注册账号并联网才能使用产品功能，数据默认存储在服务端数据库中。

### 考虑该方案的场景

- 项目初期讨论时，团队对"学习工具 + AI 增强"的定位自然联想到 SaaS 模式
- 云端优先可以快速收集用户行为数据，便于做产品迭代分析
- 市面上多数笔记/学习类产品（Notion、语雀等）均采用云端优先，属于行业主流

### 拒绝的详细理由

- **用户场景不匹配**：目标用户群体（学生）在教室、图书馆、通勤等场景中经常处于无网络或弱网络环境，云端优先无法满足离线使用需求
- **注册门槛**：用户首次接触产品就被要求注册账号，显著降低试用转化率；本地优先允许用户"先用再说"，大幅降低获客成本
- **隐私顾虑**：学生用户（尤其是未成年用户及其家长）对个人信息上传云端较为敏感，本地优先天然规避此问题
- **成本结构**：云端优先需要从一开始就投入服务器、带宽、数据库等运维成本，而 MVP 阶段应聚焦验证产品价值而非基础设施

### 未来可能的重新评估条件

- 当用户量超过一定规模（如 DAU > 5 万），本地优先架构成为维护瓶颈时
- 当产品增加强社交/协作属性（如小组学习、共享笔记库），必须依赖云端时
- 当 AI 能力必须依赖云端大模型推理且成本无法通过边缘计算解决时

## 2. 全局二元模式切换

### 方案描述

在应用顶层设置一个"本地模式 / AI 模式"的全局切换开关。用户在本地模式下完全离线、所有 AI 功能不可用；在 AI 模式下联网、所有 AI 功能激活。两种模式互斥，用户需手动切换。

### 考虑该方案的场景

- 开发实现简单——只需在顶层判断一个布尔值即可决定功能可用性
- 对用户传达清晰："你现在用的是哪种模式"
- 早期原型讨论中有人提出，便于向用户解释产品能力边界

### 拒绝的详细理由

- **体验割裂**：用户在使用过程中被迫频繁切换模式，打断学习心流。例如：正在用闪卡复习，突然想用 AI 生成摘要，必须先切换到 AI 模式，等待网络就绪
- **心智负担**：用户需要理解"模式"概念，增加了认知复杂度；好的工具应该让用户专注于学习本身，而非工具的使用方式
- **功能级按需激活更优**：采用"功能级按需激活"策略——每个功能点独立判断是否需要联网/AI，用户点击 AI 功能时才触发联网请求，其余时间完全本地运行。这种方式对用户透明，体验更自然

### 未来可能的重新评估条件

- 当产品引入大量依赖网络的实时协作功能，全局模式切换能简化状态管理时
- 当用户反馈明确表达对"功能级激活"感到困惑、希望有全局控制时
- 当合规要求（如数据出境、隐私法规）要求明确区分离线/在线状态时

## 3. CRDT 作为 MVP 同步方案

### 方案描述

采用 CRDT（Conflict-free Replicated Data Type，无冲突复制数据类型）作为多端数据同步的底层算法。CRDT 能够在无中心协调的情况下自动合并并发修改，理论上实现"最终一致性"且无数据丢失。

### 考虑该方案的场景

- 本地优先架构天然面临多设备数据同步问题，CRDT 是该领域的前沿方案
- Automerge、Yjs 等 CRDT 库已有相对成熟的 JavaScript 实现
- 长期来看，CRDT 是本地优先生态的演进方向

### 拒绝的详细理由

- **技术复杂度高**：CRDT 的数据结构设计、合并策略、垃圾回收机制均需要深入理解，团队当前无相关实战经验
- **学习曲线陡峭**：团队需要投入 2-4 周专门学习和调试 CRDT，对 MVP 紧张的工期来说不可接受
- **MVP 阶段过度设计**：MVP-1 本身是纯本地产品，不涉及同步；MVP-2 的同步需求初期仅需支持"同一用户双设备"场景，LWW（Last Write Wins）+ 版本向量方案已足够
- **调试困难**：CRDT 在边界情况下的行为难以预测，排查同步 Bug 的成本远高于简单方案

### 未来可能的重新评估条件

- 当多设备并发编辑成为高频场景（如用户在电脑和平板上同时编辑笔记）时
- 当 LWW 方案频繁出现用户可感知的数据覆盖问题，用户投诉达到一定量级时
- 当 CRDT 生态进一步成熟，出现更简单易用的封装库时
- 当团队扩充、有专人可以深入研究 CRDT 时

## 4. SQLite WASM 作为主存储

### 方案描述

使用 WebAssembly 编译的 SQLite（如 sql.js、wa-sqlite）作为浏览器端主存储引擎，替代 IndexedDB。在浏览器中运行完整的 SQLite 数据库引擎，支持完整的 SQL 查询能力。

### 考虑该方案的场景

- 团队有 SQLite 使用经验，SQL 查询比 IndexedDB 的 key-value 模式更灵活
- SQLite 的数据模型更贴合关系型业务需求（笔记-标签、闪卡-牌组等多对多关系）
- 便于未来与服务端 SQLite/PostgreSQL 做数据同步时的模型复用

### 拒绝的详细理由

- **兼容性风险**：Safari 对 WebAssembly 的 SharedArrayBuffer 和部分高级特性的支持不完整，导致 iOS/iPadOS 用户体验不可靠，而学生群体使用 iPad 的比例较高
- **IndexedDB 更通用**：IndexedDB 在所有现代浏览器中均有良好支持，是经过充分验证的浏览器存储方案
- **Dexie.js 封装足够好**：Dexie.js 为 IndexedDB 提供了类 SQL 的查询 API（支持 where、filter、sortBy 等），大幅降低了 IndexedDB 原生 API 的复杂度，开发体验接近 SQL
- **存储体积限制**：WASM 版 SQLite 需要将整个数据库加载到内存，对大型数据集（大量笔记+附件）的内存占用较高

### 未来可能的重新评估条件

- 当 Safari/WKWebView 完整支持 WASM 多线程和 SharedArrayBuffer 时
- 当业务查询需求超出 Dexie.js 能力范围（如复杂聚合、多表 JOIN）时
- 当出现成熟的 SQLite WASM 封装方案（如 Chromium 团队官方推荐方案）时
- 当本地数据量增长到 IndexedDB 性能瓶颈时（通常 > 500MB）

## 5. Flutter 跨端

### 方案描述

采用 Flutter 框架开发跨端应用，一套 Dart 代码同时编译为 Web、iOS、Android、Windows、macOS 多平台应用，替代 PWA + 响应式 Web 方案。

### 考虑该方案的场景

- 项目长期规划中明确有移动端（iOS/Android）需求
- Flutter 的跨端能力理论上可以减少多端维护成本
- 团队中有成员具备 Flutter 开发经验

### 拒绝的详细理由

- **本地优先场景 Web/PWA 更合适**：PWA 天然支持离线使用、免安装、跨平台，且 IndexedDB、Service Worker 等本地优先技术栈与 Web 深度集成
- **Flutter Web 的 Service Worker 集成不成熟**：Flutter Web 生成的 Service Worker 定制灵活度有限，难以精确控制缓存策略和离线行为
- **Flutter Web 的 IndexedDB 集成需桥接**：Dart 无法直接访问 IndexedDB，需通过 JS Interop 桥接，增加复杂度和调试难度
- **包体积问题**：Flutter Web 的初始加载包体积较大（通常 > 2MB gzip），影响首次访问体验和 SEO
- **MVP 阶段应聚焦 Web**：先做好 Web/PWA，验证产品价值后再考虑原生移动端，Flutter 可作为 V2.0 的技术选型候选

### 未来可能的重新评估条件

- 当产品验证成功、需要推出原生移动端以提升用户体验时
- 当 Flutter Web 的 WASM 渲染引擎和 Service Worker 支持显著改善时
- 当竞品推出高质量移动端，市场压力要求快速跟进时
- 当 PWA 在目标平台（尤其是 iOS）上的限制（如推送通知、后台同步）无法绕过时

## 6. 砍掉费曼模块以缩短工期

### 方案描述

将费曼学习法模块从 MVP-1 范围中移除，仅保留番茄钟、智能笔记、闪卡三大核心模块，以此缩短开发周期、降低 MVP-1 交付风险。

### 考虑该方案的场景

- 项目排期紧张时，PM 提出通过缩减范围来保证按时交付
- 费曼学习法相对独立，砍掉不影响其他模块的功能完整性
- 费曼学习法的用户认知度低于番茄钟和闪卡，被认为可能是"锦上添花"而非"雪中送炭"

### 拒绝的详细理由

- **差异化亮点**：市面上番茄钟和闪卡产品众多，费曼学习法模块是课伴的核心差异化之一，砍掉后产品同质化风险显著
- **投入产出比可接受**：MVP-1 中费曼模块仅需实现基础的四步流程（选择概念→用自己的话解释→发现知识缺口→回顾简化），工作量约 2.5 人月，在总体工期中占比可控
- **决策点保留灵活性**：在 D1 决策点（M1.5 存储评审）可以重新评估，若工期确实紧张再降级（如仅保留记录功能、去掉 AI 评估），而非一开始就砍掉
- **教育价值认同**：费曼学习法在国内外教育社区有广泛认同感，是获取种子用户和口碑传播的重要卖点

### 未来可能的重新评估条件

- D1 决策点（M1.5）时，若项目整体进度落后 > 2 周，可考虑降级费曼模块为"记录+追踪"简化版
- 若用户调研显示目标群体对费曼学习法认知度极低，可考虑替换为其他学习法模块
- 若 MVP-2 的 AI 能力足够强，可将费曼模块升级为"AI 导师对话"模式

## 7. 自建直播推流/音视频互动

### 方案描述

在产品中集成自建的直播推流和音视频互动功能，支持"在线自习室""直播授课""视频学习小组"等社交学习场景。

### 考虑该方案的场景

- 市场调研发现"在线自习室"（如 StudyWithMe 类产品）有一定用户基础
- 直播/音视频能显著提升用户粘性和在线时长
- 竞品中有产品尝试将学习工具与社交直播结合

### 拒绝的详细理由

- **偏离核心定位**：课伴的核心定位是"本地优先学习工具"，直播/音视频属于社交/内容平台属性，与工具定位冲突
- **技术栈跨度过大**：音视频涉及 WebRTC、流媒体服务器、CDN、转码等复杂技术，团队不具备相关经验，从零搭建风险极高
- **MVP 阶段不应分散精力**：MVP 的核心目标是验证"本地优先学习工具"的产品假设，音视频功能会严重分散开发资源和注意力
- **合规风险**：音视频内容涉及内容审核、未成年人保护等合规要求，增加运营复杂度和法律风险
- **成本高昂**：流媒体服务器和带宽成本远高于工具类产品，MVP 阶段不具备商业化能力来覆盖成本

### 未来可能的重新评估条件

- 当核心产品验证成功、用户量稳定增长后，可考虑接入第三方直播 SDK（如声网 Agora）做轻量级尝试
- 当用户社区中"学习小组"需求强烈且频次高时
- 当市场验证"社交学习"赛道确实存在且空间足够大时
- 当团队扩充后有专人可以负责音视频模块时

## 8. 单一 NLP 供应商锁定

### 方案描述

选择一个 NLP/AI 供应商（如 OpenAI、百度文心、讯飞星火等）作为唯一供应商，所有 AI 功能（笔记摘要、闪卡生成、费曼评估、番茄钟推荐）统一使用该供应商的 API。

### 考虑该方案的场景

- 单一供应商集成工作量最小，开发速度快
- 可以与供应商谈更好的价格折扣（承诺用量）
- 统一 API 接口减少适配层代码，降低维护成本

### 拒绝的详细理由

- **单点故障风险**：单一供应商 API 宕机或限流时，课伴所有 AI 功能全部不可用，直接影响用户体验
- **价格风险**：锁定单一供应商后，供应商涨价时缺乏议价能力，且迁移成本高
- **能力差异**：不同供应商在不同任务上的表现有差异（如 A 供应商的摘要能力强，B 供应商的评估能力强），多供应商策略可以"取各家所长"
- **合规和地缘风险**：AI 服务受政策和地区限制影响较大（如 OpenAI 在中国大陆的可用性），多供应商策略降低政策风险
- **实现成本可控**：通过抽象的 AI Provider 接口层封装，切换供应商的边际成本很低，多供应商策略的额外开发量约 1-2 人月

### 未来可能的重新评估条件

- 当某供应商在课伴所需的所有 NLP 任务上均显著领先竞品（准确率差距 > 15%）时
- 当供应商提供长期锁价合同（如 3 年固定价格）且价格低于市场价 30% 以上时
- 当团队规模小、多供应商维护成本成为实际负担时
- 当监管要求明确指定使用特定合规供应商时

## 决策总结

| 替代方案 | 核心拒绝原因 | 替代选择 |
|---------|------------|---------|
| 云端优先架构 | 不满足离线场景、注册门槛高 | 本地优先 + 云同步作为增值 |
| 全局二元模式切换 | 体验割裂、心智负担 | 功能级按需激活 |
| CRDT 作 MVP 同步 | 复杂度高、MVP 阶段过度设计 | LWW + 版本向量 |
| SQLite WASM 主存储 | 兼容性风险、内存占用高 | IndexedDB + Dexie.js |
| Flutter 跨端 | Web/PWA 更适合本地优先 | PWA + 响应式 Web |
| 砍掉费曼模块 | 差异化亮点、投入产出比可接受 | 保留，D1 可降级 |
| 自建直播/音视频 | 偏离定位、技术跨度大 | MVP 阶段不做 |
| 单一 NLP 锁定 | 单点故障、价格风险 | 多供应商 + Provider 抽象层 |
