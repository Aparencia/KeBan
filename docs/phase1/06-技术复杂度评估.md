## 一、概述

本报告对课伴（KeBan）各功能模块的技术复杂度进行系统评估，识别高风险技术点，规划预研方向，并提供初步工作量估算。评估结果将直接指导开发计划和资源分配。

**复杂度评级标准**：
- ★☆☆☆☆（1星）：常规 CRUD，无技术难点
- ★★☆☆☆（2星）：有少量技术点需关注，整体简单
- ★★★☆☆（3星）：中等复杂度，有明确技术挑战
- ★★★★☆（4星）：高复杂度，需要专项技术攻关
- ★★★★★（5星）：极高复杂度，需要独立预研和原型验证

### 2.2 详细分析

#### 2.2.1 本地存储基础设施 ★★★★☆

**复杂度来源详述**：

| 技术点 | 复杂度 | 说明 |
|--------|:------:|------|
| IndexedDB 封装层 | ★★★☆☆ | 需封装统一的 CRUD 接口、事务管理、连接池；需处理 IndexedDB 异步 API 的复杂性 |
| Schema 迁移机制 | ★★★★☆ | 需设计版本化的 Schema 管理，支持向前/向后兼容迁移，迁移失败时需回滚机制 |
| 存储容量管理 | ★★★☆☆ | 浏览器 IndexedDB 配额有限（通常磁盘的 50%），需监控用量、清理策略、超限提示 |
| 数据导入导出 | ★★★☆☆ | 三种格式（JSON/CSV/Markdown）的序列化/反序列化，大文件流式处理，导入前数据验证 |
| 全文搜索索引 | ★★★★☆ | 需在浏览器端构建倒排索引，支持中文分词，索引更新策略，1000+ 文档规模下的性能 |

**关键风险**：
- IndexedDB 在不同浏览器中的存储配额差异大（Chrome 50%、Firefox 50%、Safari 1GB）
- Schema 迁移在用户长时间不更新后一次性跨多版本迁移的边界情况
- 中文全文搜索的分词质量直接影响搜索体验

**建议方案**：
- 使用 Dexie.js 作为 IndexedDB 封装基础，降低 API 复杂度
- 采用 FlexSearch 或 MiniSearch 构建本地全文搜索索引
- Schema 迁移采用增量式（逐步版本跳跃），禁止跨版本直跳

#### 2.2.2 番茄钟 ★★☆☆☆

**复杂度来源详述**：

| 技术点 | 复杂度 | 说明 |
|--------|:------:|------|
| 计时引擎 | ★★☆☆☆ | 基于 Web Worker 实现后台计时，避免主线程阻塞；需处理 Worker 被浏览器休眠的情况 |
| Notification API | ★★☆☆☆ | 需处理权限请求、Safari 兼容性、用户拒绝后的降级方案 |
| 统计图表 | ★★☆☆☆ | 使用 ECharts 或 Chart.js 渲染柱状图+趋势线，配置工作量适中 |
| 进度环动画 | ★☆☆☆☆ | CSS/SVG 动画，技术成熟 |
| 自定义配置 | ★☆☆☆☆ | 表单 + 本地存储，常规操作 |

**关键风险**：
- Chrome 后台标签页可能降低 setTimeout/setInterval 精度，需通过 Web Worker + performance.now() 校正
- Safari 对 Notification API 的支持有限制，需提供应用内通知作为降级方案

#### 2.2.3 智能笔记（本地模式） ★★★★☆

**复杂度来源详述**：

| 技术点 | 复杂度 | 说明 |
|--------|:------:|------|
| 富文本编辑器 | ★★★★☆ | 需选型并深度定制（TipTap/ProseMirror/Slate.js），支持图片/代码块/表格，扩展性要求高 |
| 模板渲染引擎 | ★★★☆☆ | 将模板定义（JSON Schema）渲染为编辑器内容，不同模板结构差异大 |
| 三栏布局 | ★★☆☆☆ | CSS Grid/Flexbox 响应式布局，需处理各栏宽度拖拽调整 |
| 标签/文件夹系统 | ★★☆☆☆ | 树形结构管理、拖拽排序、多对多关系 |
| 版本历史 | ★★★☆☆ | 本地 diff 算法、版本快照存储、回滚操作 |
| Markdown/HTML 导入导出 | ★★★☆☆ | 富文本 ↔ Markdown 双向转换，格式保真度（表格/代码块/图片） |

**关键风险**：
- 富文本编辑器是笔记模块的核心且是最大的技术不确定性来源
- 模板渲染需要同时满足"结构约束"和"自由编辑"的平衡
- Markdown 导出的格式保真度（尤其是表格和嵌套列表）是已知难点

**建议方案**：
- 编辑器选型推荐 **TipTap**（基于 ProseMirror，API 友好，生态丰富）
- 模板使用 ProseMirror NodeSpec 定义结构约束
- Markdown 转换使用 markdown-it（解析）+ ProseMirror Markdown serializer（序列化）

#### 2.2.6 云同步引擎 ★★★★★

**复杂度来源详述**：

| 技术点 | 复杂度 | 说明 |
|--------|:------:|------|
| CRDT 数据结构 | ★★★★★ | 需要为每种数据类型选择合适的 CRDT（LWW-Register/OR-Set/RGA），设计和验证成本高 |
| 冲突检测与解决 | ★★★★★ | 多设备并发修改同一实体的冲突检测、用户友好的冲突解决 UI |
| 增量同步协议 | ★★★★☆ | 基于 OpLog 的增量推送、断点续传、批量合并优化 |
| 离线操作队列 | ★★★★☆ | append-only 操作日志、网络恢复后有序推送、队列持久化 |
| 多设备一致性 | ★★★★☆ | 保证最终一致性、处理时钟偏差、设备上下线通知 |
| 服务端架构 | ★★★★☆ | 需要搭建同步服务端、WebSocket 长连接管理、数据分片 |

**关键风险**：
- 这是整个项目技术复杂度最高的模块，建议在 MVP-2 阶段独立预研
- CRDT 选型错误可能导致后期重构成本极高
- 冲突解决的边界情况极多，需要充分的测试覆盖

**建议方案**：
- MVP-2 初期采用简化方案：OpLog + LWW（Last-Write-Wins），避免 CRDT 的复杂度
- 冲突解决 UI 采用"展示两个版本，用户选择"的简单策略
- 后期可渐进升级为 CRDT

## 三、四大预研技术点

基于复杂度评估，以下四个技术点需要在正式开发前完成预研验证：

### 预研 1：本地存储方案选型与容量评估

| 维度 | 内容 |
|------|------|
| **目标** | 确定 IndexedDB 封装库选型，评估各浏览器存储容量，制定容量管理策略 |
| **选型对比** | Dexie.js vs localForage vs 原生 IndexedDB |
| **验证内容** | ① CRUD 性能基准（1000/10000/100000 条记录） ② 各浏览器实际存储配额测试 ③ 中文全文搜索方案对比（FlexSearch vs MiniSearch vs Lunr.js） |
| **产出** | 技术选型报告 + 性能基准数据 + 容量管理方案设计 |
| **预计时间** | 1 周 |

### 预研 2：同步引擎方案验证（CRDT vs LWW）

| 维度 | 内容 |
|------|------|
| **目标** | 确定 MVP-2 同步引擎的技术方案，验证核心算法可行性 |
| **方案对比** | ① OpLog + LWW（简单方案） ② Automerge CRDT 库 ③ Yjs CRDT 库 |
| **验证内容** | ① 两种方案在文本编辑场景下的冲突解决效果 ② 离线→在线同步的性能和正确性 ③ 多设备（3+）并发场景模拟 |
| **产出** | 方案对比报告 + 原型代码 + 冲突解决策略设计 |
| **预计时间** | 2 周 |

### 预研 3：富文本编辑器选型与模板渲染

| 维度 | 内容 |
|------|------|
| **目标** | 确定笔记模块的编辑器技术栈，验证模板渲染可行性 |
| **选型对比** | TipTap (ProseMirror) vs Slate.js vs Lexical (Meta) |
| **验证内容** | ① 三种编辑器在图片/代码块/表格场景下的表现 ② 康奈尔模板（三栏结构）的实现可行性 ③ 性能（大文档编辑 > 10000 字） ④ Markdown 导入导出格式保真度 |
| **产出** | 编辑器选型报告 + 模板渲染原型 + 性能测试数据 |
| **预计时间** | 1.5 周 |

### 预研 4：SM-2 算法实现与 Anki 交叉验证

| 维度 | 内容 |
|------|------|
| **目标** | 实现 SM-2 核心算法，通过与 Anki 对比验证正确性 |
| **验证内容** | ① 使用相同的输入序列（评分序列），对比课伴 SM-2 输出与 Anki 输出 ② 边界情况测试（连续 Again、EF 下限、新卡首次复习） ③ 学习会话调度逻辑验证 |
| **产出** | SM-2 算法实现代码 + 测试用例集 + 与 Anki 对比报告 |
| **预计时间** | 0.5 周 |

| RISK-05 | 云同步冲突解决的边界情况覆盖不全 | 高 | 高 | MVP-2 初期采用 LWW 简化方案；充分测试 |
| RISK-06 | AI API 服务不稳定或延迟过高 | 中 | 中 | 多 Provider 备选；超时+重试机制；优雅降级 |
| RISK-07 | 浏览器后台限制导致计时精度下降 | 中 | 中 | Web Worker + performance.now() 校正 |
