## 一、需求基线管理

### 1.1 基线建立

| 里程碑 | 基线内容 | 建立时间 | 审批人 |
|--------|---------|---------|--------|
| **M0（需求评审通过）** | 需求规格说明书（SRS）v1.0 | 需求评审会通过后 | 产品经理 + 技术负责人 |
| **M1（需求冻结）** | MVP-1功能清单 + SRS终稿 + 用户故事地图终稿 | M1评审会通过后 | PM + 产品经理 + 技术负责人 + 客户方确认 |

### 1.2 基线管理策略

- **M0之前（需求定义阶段）**：需求可自由调整，由产品经理主导，无需走变更流程
- **M0-M1之间（需求细化阶段）**：需求调整需产品经理审批，评估对MVP范围的影响
- **M1之后（需求冻结后）**：所有需求变更必须走本文档定义的变更管理流程
- **基线版本管理**：每次基线变更必须更新版本号，保留历史基线快照
- **基线审计**：PM每月审查一次需求基线一致性，确保开发内容与基线匹配

### 1.3 基线文档清单

| 文档 | 基线版本 | 存放位置 | 更新频率 |
|------|---------|---------|---------|
| 需求规格说明书（SRS） | M1冻结版本 | 文档库/phase1/ | 仅变更时更新 |
| MVP-1功能清单 | M1冻结版本 | 项目管理工具 | 仅变更时更新 |
| 用户故事地图 | M1冻结版本 | 文档库/phase1/ | 仅变更时更新 |
| 产品原型（Figma） | M1冻结版本 | Figma（创建快照） | 仅变更时更新 |
| 验收标准 | M1冻结版本 | 项目管理工具 | 仅变更时更新 |

## 二、需求变更分类

### 2.1 变更等级定义

| 等级 | 名称 | 定义 | 审批流程 | 处理时效 | 示例 |
|------|------|------|---------|---------|------|
| **C1** | 微小变更 | 不影响功能范围和工作量 | 产品经理直接审批 | 1个工作日 | UI文案修改、非核心字段调整、错别字修正、颜色微调 |
| **C2** | 一般变更 | 影响功能细节，工作量变动≤2人天 | 产品经理+技术负责人联合评估 | 3个工作日 | 功能细节调整、交互逻辑微调、接口参数变更、边界条件补充 |
| **C3** | 重大变更 | 影响MVP范围/架构/进度/预算 | CCB（变更控制委员会）审批 | 5个工作日 | 新增功能模块、架构方向调整、MVP范围蔓延、技术栈变更 |

### 2.2 变更等级判定规则

```
变更请求进入
      ↓
是否影响MVP-1功能清单？
  ├─ 否 → 是否影响工作量 > 2人天？
  │         ├─ 否 → C1（微小变更）
  │         └─ 是 → C2（一般变更）
  └─ 是 → C3（重大变更，必须CCB审批）
```

### 2.3 CCB（变更控制委员会）组成

| 角色 | 职责 | 投票权 |
|------|------|--------|
| PM（主席） | 主持评审、综合评估、最终决策 | 1票（含否决权） |
| 产品经理 | 评估需求合理性、业务价值 | 1票 |
| 技术负责人 | 评估技术可行性、工作量、风险 | 1票 |
| 客户方代表 | 确认业务需求、审批预算/时间影响 | 1票 |

**决策规则**：4票中至少3票同意方可通过；PM拥有否决权（仅用于明显损害项目利益的情况）。

## 三、变更影响评估模板

### 3.1 四维评估框架

每个C2及以上变更必须进行以下四维评估：

| 评估维度 | 评估内容 | 评估人 | 评估标准 |
|----------|---------|--------|---------|
| **范围影响** | 变更对MVP功能清单的影响 | 产品经理 | 是否新增/删除/修改功能点；影响的用户故事数量 |
| **时间影响** | 对项目进度的影响 | PM | 是否导致里程碑延期；延期天数；是否可通过调配资源消化 |
| **成本影响** | 对项目预算的影响 | PM | 新增人力成本；新增工具/API成本；机会成本 |

## 四、MVP-1范围锁定机制

### 4.1 锁定规则

M1里程碑评审通过后，MVP-1功能清单进入锁定状态，执行以下规则：

| 规则编号 | 规则内容 | 例外情况 |
|----------|---------|---------|
| **R1** | MVP-1功能清单不可追加新功能 | CCB全票通过 + 客户方书面同意延期/增加预算 |
| **R2** | 已有功能不可升级为更高优先级 | CCB 3票以上通过 |
| **R3** | 已有功能可降级（Must→Should） | 因技术不可行或风险过高，CCB 2票以上通过 |
| **R4** | C1微小变更不受锁定限制 | 单次变更工作量不超过0.5人天 |
| **R5** | Bug修复不受锁定限制 | 确认为缺陷而非新功能 |

### 4.2 范围锁定公示

M1通过后，PM需在以下渠道公示范围锁定：
- 项目管理工具中创建"MVP-1范围锁定"公告
- 团队即时通讯群发送通知
- 邮件通知客户方接口人
- 在需求文档首页添加锁定标记

### 4.3 违反锁定的处理

- 未经CCB审批的需求追加，开发团队有权拒绝
- 追溯发现的范围蔓延，PM需在周报中标记并制定纠正措施
- 累计超过3次未审批变更，触发项目健康度审查

## 五、MVP-2 Backlog管理

### 5.1 Backlog收集机制

| 来源 | 收集方式 | 负责人 | 录入频率 |
|------|---------|--------|---------|
| 需求变更被拒绝的功能 | 变更拒绝后自动转入Backlog | 产品经理 | 实时 |
| 客户方新增需求 | 客户沟通后录入 | PM | 实时 |
| 团队内部创新想法 | 迭代回顾会收集 | 产品经理 | 每迭代 |
| 用户反馈（上线后） | 用户反馈渠道汇总 | 产品经理 | 每周 |
| 竞品新功能启发 | 竞品分析报告产出 | 产品经理 | 每月 |
| 技术债务 | 技术评审会产出 | 技术负责人 | 每迭代 |

### 5.2 Backlog条目格式

| 字段 | 说明 | 必填 |
|------|------|------|
| 编号 | BL-NNN | ✓ |
| 标题 | 简明描述 | ✓ |
| 来源 | 变更拒绝/客户新增/团队想法/用户反馈/竞品/技术债务 | ✓ |
| 优先级 | P0/P1/P2/P3（待定，MVP-2规划时重新排序） | ✓ |
| 预估工作量 | 人天（粗估） | ✗ |
| 业务价值说明 | 一句话描述价值 | ✗ |
| 提交日期 | - | ✓ |
| 状态 | 待评估/已评估/已纳入MVP-2/已放弃 | ✓ |

### 5.3 Backlog治理规则

- **定期清理**：每月审查一次Backlog，清理过时或已不适用的条目
- **优先级预排**：MVP-1开发过半时，启动MVP-2 Backlog优先级预排
- **容量规划**：MVP-2 Backlog总工作量不超过MVP-1的120%（避免无限膨胀）

## 八、变更审查周期

### 8.1 审查会议安排

| 会议类型 | 频率 | 时长 | 参与人 | 内容 |
|----------|------|------|--------|------|
| 变更审查会 | 双周一次（与迭代评审同步） | 30-60min | PM、产品经理、技术负责人 | 审查积累变更、做出决策 |
| 紧急变更评审 | 按需（紧急变更触发） | 30min | CCB全体成员 | 紧急变更的快速决策 |
| 变更回顾 | 每月一次（随月度项目回顾） | 15min | PM | 统计变更数据、分析趋势 |

### 8.2 变更审查会议议程

1. **变更清单回顾**（5min）：列出本周期内所有新增变更请求
2. **逐项评审**（每项10-15min）：
   - 提交人陈述变更内容和原因
   - 评估人报告四维评估结果
   - 讨论与投票（C3变更）
3. **决策汇总**（5min）：汇总所有决策结果
4. **行动项分配**（5min）：通过的变更分配实施负责人和排期

### 8.3 变更度量指标

| 指标 | 计算方式 | 健康阈值 | 告警阈值 |
|------|---------|---------|---------|
| 变更请求总数 | M1后累计变更数 | ≤10个/月 | >15个/月 |
| 变更通过率 | 通过数/总提交数 | ≤50% | >70% |
| C3变更占比 | C3数/总变更数 | ≤20% | >40% |
| 变更导致延期天数 | 累计因变更导致的延期天数 | ≤5天/月 | >10天/月 |
| 变更处理周期 | 提交到决策的平均天数 | ≤3个工作日 | >5个工作日 |
