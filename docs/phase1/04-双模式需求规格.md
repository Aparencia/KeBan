## 一、架构理念：功能级按需激活

### 1.1 核心原则

课伴的双模式架构采用 **"功能级按需激活"（Feature-level Progressive Enhancement）** 而非"全局二元切换"。

**用户无需理解"本地模式/AI 模式"的概念**，只需感知：
- 核心功能随时可用，无论网络状况如何
- 联网时有额外的智能功能锦上添花

这意味着"双模式"对用户是**透明的**——不存在"切换模式"的操作入口，不存在"您正在使用本地模式"的提示横幅。用户感知到的是**功能的自然丰富或减少**，而非模式的二元跳变。

### 1.2 设计哲学

| 反模式（避免） | 正确做法 |
|-------------|---------|
| 全局模式开关："切换到离线模式" | 功能级检测：每个 AI 功能独立检测可用性 |
| 断网时弹出"功能不可用"全屏警告 | AI 按钮点击时温和提示"此功能需要网络" |
| 联网时强制登录才能使用基础功能 | 基础功能零登录可用，AI 功能按需登录 |
| UI 因网络状态变化大面积重构 | UI 保持稳定，AI 功能按钮状态动态变化 |

### 1.3 三层能力模型

```
┌──────────────────────────────────────────┐
│        第三层：AI 增强能力                  │
│  （智能摘要·AI生卡·解释评估·语义搜索）        │
│  ── 需要：网络 + AI API 可用               │
├──────────────────────────────────────────┤
│        第二层：云端协作能力                  │
│  （云同步·多设备·云备份·用户账户）            │
│  ── 需要：网络可用                          │
├──────────────────────────────────────────┤
│        第一层：本地核心能力                  │
│  （计时·笔记·闪卡·费曼·统计·搜索·导出）       │
│  ── 需要：无（纯浏览器环境）                 │
└──────────────────────────────────────────┘
```

## 三、模式切换用户体验设计

### 3.1 AI 功能按钮交互规范

所有 AI 增强功能在 UI 上**始终显示**对应按钮/入口，不因网络状态隐藏。交互规范如下：

```
┌─────────────────────────────────────────────────────┐
│  AI 功能按钮交互状态机                                 │
│                                                      │
│  ┌──────────┐    点击     ┌──────────────┐           │
│  │ 按钮可用  │ ─────────▶ │ 检测可用性    │           │
│  │（默认态） │            └──────┬───────┘           │
│  └──────────┘                   │                    │
│                           ┌─────┴─────┐             │
│                           ▼           ▼             │
│                    ┌──────────┐ ┌───────────┐       │
│                    │ 可用：    │ │ 不可用：   │       │
│                    │ 执行功能  │ │ 显示提示   │       │
│                    └──────────┘ └───────────┘       │
└─────────────────────────────────────────────────────┘
```

### 3.2 可用性检测流程

用户点击 AI 功能按钮时，系统执行以下检测链：

1. **网络连通性检测**（< 500ms）
   - 发送轻量 ping 请求（HEAD /api/health）
   - 超时或失败 → 提示"需要网络连接"

2. **AI 服务可用性检测**（< 1s）
   - 检查 AI API 端点是否响应正常
   - 检查用户账户是否有 AI 功能权限
   - 失败 → 提示"AI 服务暂时不可用，请稍后重试"

3. **功能执行**
   - 通过检测 → 正常调用 AI 功能
   - 显示加载动画（骨架屏或进度条）

### 3.3 不可用时的友好提示设计

| 场景 | 提示文案 | 替代操作 |
|------|---------|---------|
| 无网络连接 | "此功能需要网络连接。您的笔记已安全保存，联网后可使用 AI 摘要功能。" | 提供"手动创建摘要"入口 |
| AI 服务繁忙 | "AI 助手正忙，请稍后重试。您也可以先手动整理笔记要点。" | 提供"稍后提醒我"选项 |
| AI 服务异常 | "AI 功能暂时不可用，工程师正在修复中。您的数据不受影响。" | 提供问题反馈入口 |
| 未登录/无权限 | "AI 增强功能需要登录使用。点击下方登录解锁智能摘要等功能。" | 提供登录入口，基础功能不受影响 |

### 3.4 网络状态指示器

在应用顶栏右侧设置一个**低调但可见**的网络状态指示器：

- 🟢 绿色小圆点：在线 + AI 可用
- 🟡 黄色小圆点：在线但 AI 不可用
- 🔴 灰色小圆点：离线
- 悬停显示详细说明文字
- 不弹 toast，不打断用户操作

## 四、降级策略

### 4.1 渐进式降级原则

**核心原则**：渐进式降级（Progressive Degradation），而非硬切换（Hard Switch）。

| 降级触发场景 | 降级方式 | 用户感知 |
|------------|---------|---------|
| 使用过程中断网 | 已打开的 AI 面板保持显示，新请求失败时温和提示 | 不关闭当前面板，不重置编辑内容 |
| 应用启动时无网络 | 正常加载所有本地功能，AI 按钮显示为"需要网络"状态 | 无报错，无空白页 |
| AI 请求超时（> 10s） | 取消请求，提示超时，保留用户已输入的内容 | 不丢失用户操作上下文 |
| AI 返回异常内容 | 显示"AI 结果可能不准确"警告，仍展示结果供用户判断 | 不直接丢弃结果 |

### 4.2 乐观更新策略

**原则**：先本地生效，后台同步（Optimistic Update）。

```
用户操作 → 立即更新本地 UI → 写入 IndexedDB → 后台尝试同步到云端
                                   │
                            ┌──────┴──────┐
                            ▼             ▼
                       同步成功        同步失败
                       (静默)      (加入重试队列)
                                      │
                                 网络恢复后批量推送
```

**具体规则**：
- 所有数据写入操作**首先写入本地 IndexedDB**，UI 立即反映变更
- 联网状态下，后台异步将变更推送到云端
- 推送失败时，将操作记录加入本地队列（append-only 操作日志）
- 网络恢复后，按顺序批量推送队列中的操作

### 4.3 功能降级矩阵

| AI 功能 | 降级后替代方案 | 实现方式 |
|---------|-------------|---------|
| AI 智能摘要 | 手动要点提取工具 | 提供"要点列表"模板，用户手动填写 |
| AI 续写/润色 | 常规编辑功能 | 编辑器本身功能不受影响 |
| AI 关键词提取 | 手动高亮标记 | 提供文本选中高亮工具（颜色标注） |
| AI 语义搜索 | 关键词全文搜索 | 回退到本地倒排索引搜索 |
| AI 自动生成卡片 | 手动创建卡片 | 提供快速创建模板和批量导入入口 |
| AI 解释质量评估 | 自我评估清单 | 展示费曼自评检查清单 |
| AI 引导问题 | 空白编辑区 | 显示"尝试从以下角度思考"的通用提示 |
| AI 录音转文字 | 手动文字输入 | 编辑器正常可用 |

## 五、数据同步规格

### 5.1 同步架构

```
┌─────────────┐         ┌─────────────┐         ┌─────────────┐
│   设备 A     │         │   云端服务    │         │   设备 B     │
│             │         │             │         │             │
│  IndexedDB  │ ──────▶ │  操作日志    │ ◀────── │  IndexedDB  │
│  操作日志    │         │  (append-only)│        │  操作日志    │
│             │ ◀────── │             │ ──────▶ │             │
└─────────────┘         └─────────────┘         └─────────────┘
```

### 5.2 同步协议

#### 操作日志格式（OpLog）

每条本地操作记录以下字段：

```json
{
  "opId": "uuid-v4",
  "deviceId": "device-uuid",
  "timestamp": "ISO-8601",
  "entityType": "note|card|deck|pomodoro|feynman",
  "entityId": "entity-uuid",
  "operation": "create|update|delete",
  "payload": { ... },
  "version": 42
}
```

#### 同步流程

1. **本地操作**：用户操作 → 生成 OpLog → 写入本地 IndexedDB → UI 立即更新
2. **后台推送**：
   - 联网状态下，后台 Worker 定期（30 秒）或实时推送未同步的 OpLog
   - 推送前按 entityType + entityId 合并连续操作（减少网络请求）
3. **云端接收**：
   - 云端验证 OpLog 合法性
   - 与云端最新版本比对
   - 无冲突 → 直接应用
   - 有冲突 → 进入冲突解决流程

### 5.3 冲突解决策略

**原则**：保留双方版本，不丢弃任何用户数据。

| 冲突类型 | 解决策略 | 说明 |
|---------|---------|------|
| 同一字段并发修改 | 保留双方版本（创建分叉） | 用户下次同步时看到两个版本，选择保留或合并 |
| 同一笔记一方删除一方修改 | 保留修改方 | 删除意图弱于修改意图 |
| 同一卡片双方删除 | 确认删除 | 删除意图一致，直接执行 |
| 牌组结构冲突 | 合并（取并集） | 双方新增的子牌组都保留 |

### 5.4 离线操作队列

当设备离线时：

1. 所有操作正常执行，写入本地 IndexedDB
2. 同步模块将操作追加到本地 OpLog 队列（append-only）
3. 网络恢复时：
   - 检测网络连通性（连续 3 次 ping 成功）
   - 从云端拉取最新状态（lastSyncTimestamp）
   - 比对本地 OpLog 中 timestamp > lastSyncTimestamp 的操作
   - 按时间顺序批量推送到云端
   - 推送过程中处理冲突

### 5.5 同步状态展示

| 状态 | 图标 | 说明 |
|------|------|------|
| 已同步 | ✅ | 所有本地数据已与云端一致 |
| 同步中 | 🔄 | 正在推送/拉取数据 |
| 待同步（N 条） | ☁️ | 有 N 条本地操作尚未同步 |
| 同步冲突 | ⚠️ | 需要用户介入解决冲突 |
| 离线 | ⭕ | 无网络连接，操作已暂存本地 |

## 六、安全与隐私规格

### 6.1 本地数据安全

| 措施 | 说明 |
|------|------|
| IndexedDB 加密 | 敏感字段使用 AES-256 加密存储，密钥由用户密码派生（PBKDF2） |
| 操作日志签名 | OpLog 每条记录带 HMAC 签名，防止篡改 |
| 本地缓存清理 | 浏览器缓存中不存储明文笔记内容 |

### 6.2 传输安全

| 措施 | 说明 |
|------|------|
| HTTPS 强制 | 所有云端通信使用 HTTPS（TLS 1.3） |
| 端到端加密（远期） | 笔记和卡片内容端到端加密，服务端不存储明文 |
| Token 刷新 | JWT 短期 Token + Refresh Token 机制 |

## 七、性能规格

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 网络可用性检测 | < 500ms | HEAD 请求 ping |
| AI 功能降级响应 | < 200ms | 检测到不可用后立即展示替代方案 |
| 离线→在线同步启动 | < 2s | 网络恢复后 2 秒内开始推送队列 |

| 冲突检测 | < 1s | 云端比对并返回冲突结果 |
| 批量同步（100 条 OpLog） | < 10s | 网络恢复后批量推送 |

## 八、测试验收要点

| 场景 | 预期结果 |
|------|---------|
| 完全断网使用 30 分钟 | 所有本地功能正常，无报错弹窗，数据完整保存 |
