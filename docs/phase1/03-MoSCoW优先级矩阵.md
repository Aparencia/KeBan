# 课伴 MoSCoW 功能优先级矩阵

> **版本**：v1.0  
> **阶段**：Phase 1 — 需求阶段  
> **最后更新**：2026-07-08

---

## 概述

本文档使用 MoSCoW 方法（Must have / Should have / Could have / Won't have）对课伴所有功能需求进行优先级分级，指导迭代交付节奏。

**版本规划对应**：
- **Must have** → MVP-1 本地核心版（核心产品，无网络依赖）
- **Should have** → MVP-2 AI 增强版（AI 能力叠加 + 云同步）
- **Could have** → V2.0 远期版（差异化竞争力功能）
- **Won't have** → 暂不纳入（非核心赛道，资源聚焦）

---

## 一、Must Have — MVP-1 本地核心版

> **原则**：所有功能在纯离线环境下完整可用，不依赖任何网络服务或 AI 接口。这是课伴的"地基"，必须首先交付。

### 1.1 基础设施层

| # | 功能项 | 优先级理由 |
|---|--------|----------|
| INF-01 | IndexedDB 封装层 | 本地优先架构的基石，所有模块数据持久化依赖此层；包含 Schema 定义、CRUD 封装、事务管理 |
| INF-02 | Schema 迁移机制 | 支持应用升级时数据库结构自动迁移，避免用户数据丢失 |
| INF-03 | 数据导入导出（JSON/CSV/Markdown） | 用户对数据掌控权的基本保障；也是数据备份和迁移的唯一途径（无云同步时） |
| INF-04 | 基础本地账户系统 | 无需网络注册，本地 Profile 管理（用户名/头像/偏好），为后续多用户和多设备打基础 |
| INF-05 | 学习数据基础看板 | 跨模块汇总展示：今日专注时长、笔记数、复习卡片数、费曼完成数，给用户即时价值感 |

### 1.2 番茄钟模块

| # | 功能项 | 优先级理由 |
|---|--------|----------|
| PT-M01 | 基础计时引擎（工作/短休息/长休息三段式） | 番茄钟的核心价值，无此功能模块无存在意义 |
| PT-M02 | 上课模式 + 自习模式 | 区分课伴与普通计时器的关键差异点，直接对应核心用户场景（网课 vs 自习） |
| PT-M03 | 每日专注统计 | 量化学习投入的基础能力，是用户持续使用的核心动力之一 |
| PT-M04 | 自定义配置（时长/提示音） | 不同课程时长不同，固定配置无法满足真实使用需求 |
| PT-M05 | Notification API + 声音提醒 | 计时结束必须有有效通知手段，否则用户需频繁手动查看 |
| PT-M06 | 本地数据持久化 | 统计功能的前提，无持久化则历史数据无从谈起 |

### 1.3 智能笔记模块

| # | 功能项 | 优先级理由 |
|---|--------|----------|
| NT-M01 | 富文本编辑器（标题/列表/代码块/表格/图片） | 笔记工具的基本功，不具备则无法作为主力笔记工具 |
| NT-M02 | 3-5 个内置模板（大纲式/康奈尔/思维导图/自由/速记） | 课伴笔记模块的核心差异化，直接提升笔记结构化程度 |
| NT-M03 | 本地自动保存 + 持久化 | 笔记数据安全性底线，防止意外丢失 |
| NT-M04 | 全文搜索（本地索引） | 笔记量增长后的必备检索能力，否则知识库沦为"信息黑洞" |
| NT-M05 | 标签/文件夹组织系统 | 笔记分类管理的基础架构，随笔记数量增长价值愈发显著 |
| NT-M06 | 导入导出（Markdown/HTML/JSON） | 数据自由流动保障，用户进入和迁出无门槛 |

### 1.4 闪卡模块

| # | 功能项 | 优先级理由 |
|---|--------|----------|
| FC-M01 | 卡片 CRUD（正面+背面，文字/图片/代码） | 闪卡系统的基本载体，无卡片则无后续一切 |
| FC-M02 | SM-2 调度引擎（Again/Hard/Good/Easy） | 间隔重复算法是闪卡模块的核心科学价值，是用户选择课伴而非普通卡片的根本原因 |
| FC-M03 | 学习会话管理（新卡+复习混合推送） | 将算法落地的交互形式，用户通过会话完成每日复习 |
| FC-M04 | 牌组管理（含子牌组嵌套） | 知识分类组织的必要能力，尤其对多科目用户至关重要 |
| FC-M05 | 基础学习统计（今日复习数/正确率） | 给用户学习进度反馈，维持持续复习动力 |

### 1.5 费曼学习法模块

| # | 功能项 | 优先级理由 |
|---|--------|----------|
| FM-M01 | 四步流程引导引擎 | 费曼模块的核心功能，流程引擎是整个模块的骨架 |
| FM-M02 | 文字记录（大面积编辑区） | 步骤 2 和步骤 4 的核心交互，无此则无法完成"用自己的话解释" |
| FM-M03 | 薄弱点标记（高亮+归入待补强列表） | 费曼学习法的核心价值在于发现知识漏洞，薄弱点标记是实现此目标的关键功能 |
| FM-M04 | 进度追踪（完成状态+理解深度自评） | 帮助用户追踪自己的学习轨迹，维持长期使用动力 |

## 二、Should Have — MVP-2 AI 增强版

> **原则**：在 MVP-1 稳固基础上叠加 AI 能力和云同步，大幅提升产品竞争力。需网络连接，但所有 AI 功能必须有离线降级策略。

### 2.1 云同步与账户

| # | 功能项 | 优先级理由 |
|---|--------|----------|
| SYNC-S01 | 云同步引擎（设备间增量同步） | 多设备用户的刚性需求；技术复杂度高，需要专项研发 |
| SYNC-S02 | 冲突解决机制（保留双方版本） | 多设备同时操作必然产生冲突，无解决机制则数据可能丢失 |
| SYNC-S03 | 云用户系统（注册/登录/多设备管理） | 云同步的身份基础，支持手机号/邮箱注册 |
| SYNC-S04 | 桌面端（Electron 封装） | 学生网课主要使用电脑，桌面端提供更好体验（通知、后台运行） |

### 2.2 AI 增强功能

| # | 功能项 | 优先级理由 |
|---|--------|----------|
| AI-S01 | AI 笔记智能摘要（US-NT-3） | 高频刚需功能，课后复习场景下价值极高；技术上对接 LLM API 相对成熟 |
| AI-S02 | AI 自动生成闪卡（US-FC-3） | 打通"笔记→卡片"的自动化链路，大幅降低制卡成本 |
| AI-S03 | AI 费曼解释质量评估 | 费曼学习法最大的痛点是"不知道自己解释得对不对"，AI 评估直接解决此问题 |
| AI-S04 | AI 番茄钟智能时长推荐 | 基于数据的个性化推荐，提升计时效率；相对独立，可增量上线 |
| AI-S05 | AI 笔记续写/润色 | 辅助写作能力，对表达能力较弱的学生帮助明显 |
| AI-S06 | AI 关键词提取高亮 | 自动识别重点内容，提升复习效率 |

### 2.3 框架能力

| # | 功能项 | 优先级理由 |
|---|--------|----------|
| FW-S01 | 双模式切换框架（运行时网络检测） | AI 功能的运行基础，必须保证"联网时启用 AI，断网时优雅降级" |
| FW-S02 | 功能渐进式降级/升级 | 用户体验一致性保障，不因网络状态突变导致界面跳动或功能中断 |
| FW-S03 | 可插拔 AI Provider 架构 | 避免锁定单一 AI 服务商，支持灵活切换（OpenAI/Anthropic/本地模型） |

## 三、Could Have — V2.0 远期版

> **原则**：差异化竞争力功能，需要前期技术储备和市场验证后再投入。

| # | 功能项 | 优先级理由 |
|---|--------|----------|
| CV-01 | 笔记语义搜索（向量数据库） | 比关键词搜索更精准，但需要向量存储基础设施，技术门槛较高 |
| CV-02 | 知识图谱自动生成 | 基于笔记内容自动构建知识点关联图谱，极具差异化价值，但 NLP 技术复杂度高 |
| CV-03 | 小组协作笔记与闪卡分享 | 协作学习场景扩展，需要实时同步和权限管理体系 |
| CV-04 | 家长端监控看板 | 面向家长的学习进度可视化，满足家长关切但非核心用户需求 |
| CV-05 | 模板市场（UGC） | 用户自建模板分享/交易，需要内容审核和经济系统 |
| CV-06 | 移动端 App（iOS/Android） | 拓展使用场景，但开发和维护成本大幅增加 |
| CV-07 | 与学校教务系统对接 | 自动同步课程表和作业，但对接标准不统一，推广难度大 |
| CV-08 | 学习社区（匿名讨论） | 学生间学习经验交流，需要运营和内容审核 |
| CV-09 | 游戏化激励系统（成就/排行榜） | 提升用户粘性和学习动力，但设计不当可能引发负面竞争 |

## 四、Won't Have — 暂不纳入

> **原则**：非核心赛道功能，与课伴"个人学习辅助工具"的定位不符，或资源投入产出比极低。

| # | 功能项 | 排除理由 |
|---|--------|----------|
| WN-01 | 自建直播推流平台 | 课伴是学习辅助工具而非在线教育平台，已有腾讯会议/钉钉等成熟方案，重复建设无意义 |
| WN-02 | 实时音视频互动答疑 | 同上，音视频互动属于教学平台范畴，非课伴核心能力圈 |
| WN-03 | 付费课程交易系统 | 涉及支付、版权、合规等复杂问题，与工具型产品定位严重偏离 |
| WN-04 | 作业批改系统 | 需要题库建设和学科知识图谱，工作量巨大且非课伴核心价值 |
| WN-05 | 线下课堂场景支持 | 课伴定位为网课辅助，线下场景的交互需求（如扫码签到）与产品架构不符 |
| WN-06 | 实时协同编辑（多人同时编辑同一文档） | 技术复杂度极高（OT/CRDT），且非学生用户核心需求 |
| WN-07 | 自建 LLM 模型 | 需要大量算力和训练数据，远超出工具型产品的能力边界 |

## 七、决策记录

| 决策 | 理由 |
|------|------|
| MVP-1 不含云同步 | 聚焦核心价值验证，避免分散资源在复杂基础设施上 |
| MVP-1 不含任何 AI 功能 | 确保产品在最简架构下可用，AI 作为增值层后续叠加 |
| 双模式框架放入 MVP-2 | 无 AI 功能时不需要此框架，避免过早引入复杂度 |
| Electron 放入 MVP-2 | 优先验证 Web 端产品市场契合度，再投入桌面端开发 |
| 移动端放入 V2.0 | 学生网课主要使用电脑，移动端优先级低于桌面端 |
| 不含直播/音视频功能 | 严格守住"学习辅助工具"边界，不越界到教学平台领域 |
