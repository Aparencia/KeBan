# 课伴项目贯穿全周期的关键管理活动

本文档定义课伴项目从启动到上线全生命周期中需持续执行的管理活动，确保项目质量、进度和风险可控。

---

## 一、文档管理

### 1.1 文档统一存放与分类编号

所有项目文档统一存放于项目根目录 `docs/` 下，按阶段和类型分类：

```
docs/
├── phase0/          # P0 需求阶段文档
│   ├── PRD/         # 产品需求文档
│   └── 调研/        # 市场调研、竞品分析
├── phase1/          # P1 设计阶段文档
│   ├── 架构/        # 技术架构文档
│   ├── UI/          # UI/UX 设计稿
│   └── 接口/        # 接口设计文档
├── phase2/          # P2 开发阶段文档
│   ├── 开发规范/    # 编码规范、代码审查标准
│   └── 测试/        # 测试计划、测试用例
├── phase3/          # P3 上线阶段文档
│   ├── 部署/        # 部署手册、运维文档
│   └── 发布说明/    # Release Notes
├── templates/       # 文档模板
├── 会议纪要/        # 各类会议记录
└── 管理文档/        # 项目管理相关文档
```

**编号规则**：`[阶段]-[类型]-[序号]-[版本]`，例如 `P0-PRD-001-v1.0`

### 1.2 版本号管理

- **大版本**（v1.0、v2.0）：里程碑节点交付，经评审通过的正式版本
- **小版本**（v1.0 → v1.1）：内容修订、错误修正
- **草稿版**（v0.x）：未定稿的工作中文档
- 所有正式文档需包含版本历史表：

| 版本号 | 修改日期 | 修改人 | 修改内容 |
|-------|---------|-------|---------|
| v1.0  | 2026-07-08 | XXX | 初始版本 |

### 1.3 文档变更留痕

- 所有文档纳入 Git 版本控制，每次修改需提交 commit 并写明变更原因
- 重要文档（PRD、架构设计、接口文档）的变更需通过 PR Review 流程
- 保留历史版本，不得直接覆盖；重大变更需在文档末尾"变更记录"章节登记

### 1.4 敏感文档权限管理

| 文档类型 | 访问权限 | 说明 |
|---------|---------|------|
| PRD、技术架构 | 全员可见 | 产品和技术核心文档 |
| 用户数据分析 | 仅 PM + 数据分析师 | 涉及用户隐私 |
| 商业计划书 | 仅管理层 | 商业机密 |
| 接口文档（内部） | 开发团队 | 安全敏感 |
| 会议纪要 | 参会人员 + 管理层 | 按需开放 |

### 1.5 文档审查机制

- 每里程碑节点进行一次文档完整性审查
- 新成员入职时进行文档导航培训
- 文档过期超过 3 个月未更新的，自动标记为"待审查"

## 二、风险管理

### 2.1 风险登记册

维护一份动态更新的风险登记册（存放于 `docs/管理文档/风险登记册.md`），每条风险包含以下字段：

| 字段 | 说明 |
|-----|------|
| 风险 ID | R-[序号]，如 R-001 |
| 风险描述 | 具体风险内容 |
| 类别 | 技术/产品/商业/人员/外部 |
| 概率 | 低/中/高 |
| 影响 | 低/中/高 |
| 风险等级 | 概率 × 影响（绿/黄/红） |
| 应对策略 | 规避/转移/缓解/接受 |
| 责任人 | 指定 owner |
| 触发条件 | 何时启动应急计划 |
| 状态 | 开放/监控中/已关闭 |

### 2.2 风险审查周期

- **每两周更新一次**风险登记册，由项目经理主持
- 每次迭代评审会（Sprint Review）中包含风险审查议程
- 每里程碑进行一次全面风险复盘

### 2.3 三大重点风险

#### 风险一：本地存储容量限制（R-001）

- **描述**：浏览器 IndexedDB 存储配额通常为磁盘空间的 50-80%，但部分浏览器/系统可能设置更低上限
- **概率**：中 | **影响**：高 | **等级**：黄
- **应对策略**：
  - M1.5 存储评审中验证各浏览器实际配额
  - 实现存储使用量监控，接近阈值时提示用户清理
  - 设计附件压缩和清理机制
  - 预留"存储迁移到 OPFS"的技术方案

#### 风险二：同步冲突处理（R-002）

- **描述**：MVP-2 云同步上线后，多设备并发修改导致数据冲突
- **概率**：中 | **影响**：中 | **等级**：黄
- **应对策略**：
  - MVP-2 初期采用 LWW（Last Write Wins）+ 版本向量
  - 冲突发生时保留冲突副本，让用户手动选择
  - 后续迭代引入字段级合并策略
  - 记录冲突日志用于分析用户行为

#### 风险三：市场窗口（R-003）

- **描述**：本地优先学习工具赛道可能被竞品抢先占领
- **概率**：低 | **影响**：高 | **等级**：黄
- **应对策略**：
  - MVP-1 严格控制范围，确保 W28 按时发布
  - 提前启动社区运营和内容营销
  - 通过 Beta 测试收集早期用户反馈，快速迭代

### 2.4 风险审查会议流程

```
1. 回顾上周新增/关闭的风险项
2. 评估现有风险的概率/影响是否有变化
3. 讨论风险应对措施的执行进展
4. 识别新出现的风险
5. 分配新风险的责任人
6. 更新风险登记册
```

### 2.5 风险升级机制

| 风险等级 | 处理层级 | 升级条件 |
|---------|---------|---------|
| 绿色（低×低） | 开发团队自行处理 | 升级为黄色时 |
| 黄色（中×中） | 项目经理跟踪 | 触发条件达到或升级为红色时 |
| 红色（高×高） | 管理层决策 | 触发应急预案时 |

- 红色风险需在 **24 小时内**召开紧急会议
- 黄色风险需在 **一周内**制定应对计划
- 所有已关闭风险需进行事后复盘

## 三、需求变更管理

### 3.1 变更管理原则

- **M1（需求冻结）前**：需求可自由讨论和调整
- **M1 后**：所有需求变更需经 CCB（Change Control Board，变更控制委员会）审批
- **MVP-1 范围锁定后**：不可追加任何新功能，所有新需求进入 MVP-2 Backlog
- 变更不是"不能做"，而是要"评估清楚再做"

### 3.2 变更流程五步法

```
┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐
│ 提交  │──▶│ 评估  │──▶│ 审批  │──▶│ 实施  │──▶│ 验证  │
│      │   │      │   │      │   │      │   │      │
│提出方 │   │技术+PM│   │ CCB  │   │开发团队│   │QA+PM │
└──────┘   └──────┘   └──────┘   └──────┘   └──────┘
```

**第一步：提交**
- 任何干系人均可提交变更请求（CR）
- 填写变更请求表：变更内容、原因、期望上线时间

**第二步：评估**
- 技术负责人评估开发工作量（人天）
- PM 评估对产品范围、进度、成本的影响
- 输出《变更影响评估报告》

**第三步：审批**
- CCB 成员（项目经理 + 技术负责人 + PM + 客户代表）投票决策
- 审批结果：批准 / 拒绝 / 延期 / 需补充信息
- MVP-1 锁定后的变更默认拒绝，除非存在重大业务风险

**第四步：实施**
- 纳入最近迭代或创建专项迭代
- 指定开发负责人和测试负责人
- 更新相关文档

**第五步：验证**
- QA 执行回归测试
- PM 验收变更效果
- 更新需求文档和变更记录

### 3.3 变更预算预留

- 总工期预留 **10-15%** 作为变更缓冲，约 **5-8 人月**
- 变更消耗超过预算 80% 时触发预警
- 变更预算未使用部分可用于技术债务清理或体验优化

### 3.4 双周变更审查

- 每两周召开一次变更审查会
- 回顾两周内的变更请求处理情况
- 统计变更趋势：若变更频率持续走高，需检查需求定义质量
- 变更请求积压超过 10 条时，优先安排集中评审

### 3.5 新需求管理

- MVP-1 开发期间产生的所有新需求统一录入 **MVP-2 Backlog**
- 每个需求标注优先级（P0/P1/P2/P3）和粗略工作量估算
- MVP-2 规划时从 Backlog 中按优先级选取
- 定期（每月一次）清理过时需求

## 四、质量保障

### 4.1 代码审查

- **每次 PR 至少 1 人 Review**，核心模块（存储、同步、AI）需 2 人 Review
- Review 重点：逻辑正确性、边界条件、错误处理、安全漏洞、性能影响
- 代码审查标准文档存放于 `docs/phase2/开发规范/`
- PR 超过 48 小时未 Review 的自动提醒 Reviewer
- 审查发现的问题分为：Must Fix（必须修复）/ Should Fix（建议修复）/ Nitpick（锦上添花）

### 4.2 自动化测试

| 测试类型 | 覆盖范围 | 工具 | 运行时机 |
|---------|---------|------|---------|
| 单元测试 | 核心业务逻辑、工具函数 | Jest / Vitest | 每次提交 |
| 集成测试 | 模块间交互、API 调用 | Jest / Vitest | 每次 PR |
| E2E 测试 | 关键用户流程 | Playwright / Cypress | 每次部署到 Staging |
| 性能测试 | 首屏加载、大数据量操作 | Lighthouse / 自定义脚本 | 每周一次 |
| 安全测试 | XSS、CSRF、数据注入 | OWASP ZAP / 手动审计 | 每里程碑 |

### 4.3 离线场景专项测试

离线场景是本项目的核心差异化，需作为专项进行测试：

- **首次离线测试**：全新设备无网络环境下完成核心流程（番茄钟、笔记、闪卡）
- **断网恢复测试**：使用中断网后恢复，验证数据不丢失、状态正确恢复
- **弱网测试**：3G/慢速 Wi-Fi 环境下的功能可用性
- **长期离线测试**：连续离线 7 天后数据完整性验证
- **存储满测试**：IndexedDB 接近配额上限时的降级处理
- 使用 Chrome DevTools 的 Network Throttling 和 Offline 模式辅助测试

### 4.4 AI 效果定期评估

- **评估周期**：每两周一次，每次评估 50+ 条 AI 生成结果
- **评估维度**：

| 维度 | 指标 | 目标 |
|-----|------|------|
| 准确性 | 生成内容与原文的语义一致性 | ≥ 85% |
| 完整性 | 关键信息是否遗漏 | ≥ 90% |
| 可用性 | 生成内容是否无需大幅修改即可使用 | ≥ 75% |
| 响应时间 | AI API 平均响应时间 | ≤ 3 秒 |
| 成本 | 单次调用平均成本 | 控制在预算内 |

- 评估结果记录于《AI 效果评估报告》，存入 `docs/管理文档/`
- 准确率低于目标时触发 Prompt 优化或供应商切换讨论

### 4.5 测试覆盖率目标

| 阶段 | 单元测试覆盖率 | 集成测试覆盖 | E2E 覆盖 |
|-----|-------------|------------|---------|
| MVP-1 Alpha | ≥ 60% | 核心模块 | 核心流程 |
| MVP-1 Release | ≥ 75% | 全部模块 | 全部核心流程 |
| MVP-2 Alpha | ≥ 65% | 核心模块 + 同步模块 | 核心流程 + 同步流程 |
| MVP-2 Release | ≥ 80% | 全部模块 | 全部流程 |
| 正式上线 | ≥ 85% | 全部模块 | 全部流程 + 回归 |

## 五、验收标准

### 5.1 阶段交付物验收

每个里程碑/阶段结束时，交付物需经客户（或客户代表）签字确认：

| 阶段 | 交付物 | 验收方 | 验收形式 |
|-----|-------|-------|---------|
| M1 需求冻结 | PRD、需求清单 | 客户 + PM | 评审会 |
| M1.5 存储评审 | 存储方案评估报告 | 技术委员会 | 技术评审 |
| M2 预研完成 | 预研报告、技术选型文档 | 技术负责人 + PM | 评审会 |
| M3 设计冻结 | UI 设计稿、接口文档 | 客户 + 开发 | 评审会 |
| MVP-1 Alpha | 可运行的本地学习工具 | 客户 + QA | 演示 + 试用 |
| MVP-1 Release | 经测试的完整版本 | 客户 | 正式验收 |
| MVP-2 Alpha | 云同步 + AI 增强版本 | 客户 + QA | 演示 + 试用 |
| MVP-2 Release | 经测试的完整版本 | 客户 | 正式验收 |
| Beta 公测 | 面向真实用户的版本 | 用户 + PM | 公测反馈 |
| 正式上线 | 生产环境版本 | 客户 + 运维 | 上线确认 |

### 5.2 MVP-1 与 MVP-2 独立验收

- MVP-1 和 MVP-2 作为两个独立可交付成果，各自有独立的验收标准和验收会议
- MVP-1 验收不依赖 MVP-2 的任何功能
- MVP-2 验收包含"与 MVP-1 的集成测试"

### 5.3 验收清单

每个 MVP 验收前，使用验收清单逐项打钩确认：

**MVP-1 验收清单示例**：
- [ ] 番茄钟：基础计时功能正常
- [ ] 番茄钟：上课/自习模式可切换
- [ ] 番茄钟：统计数据正确展示
- [ ] 智能笔记：富文本编辑功能完整
- [ ] 智能笔记：模板功能可用
- [ ] 智能笔记：搜索功能正确
- [ ] 闪卡：CRUD 操作正常
- [ ] 闪卡：SM-2 算法正确执行
- [ ] 闪卡：学习会话可正常进行
- [ ] 费曼学习法：四步流程可完整执行
- [ ] 本地存储：数据持久化正常
- [ ] 本地存储：导入导出功能正确
- [ ] 离线可用性：无网络环境下核心功能全部可用
- [ ] 性能：首屏加载 ≤ 3 秒（4G 网络）
- [ ] 兼容性：Chrome/Safari/Firefox/Edge 最新版正常运行

### 5.4 验收问题分类

| 分类 | 定义 | 处理方式 |
|-----|------|---------|
| **Blocker** | 核心功能不可用、数据丢失、崩溃 | 必须修复后才可验收 |
| **Critical** | 重要功能异常但有 workaround | 必须修复后才可验收 |
| **Major** | 功能基本可用但体验差 | 可在下一迭代中修复，但需记录承诺 |
| **Minor** | UI 细节、文案错误等 | 可在后续迭代中解决 |

- Blocker 和 Critical 问题的数量必须为 **0** 才能通过验收
- Major 问题数量 ≤ 5 个可接受，但需附带修复计划
- Minor 问题不阻塞验收，但需录入 Backlog

## 六、沟通机制

### 6.1 日常沟通

| 沟通活动 | 频率 | 时长 | 参与人 | 目的 |
|---------|------|------|-------|------|
| 每日站会 | 每日 | ≤ 15 分钟 | 全开发团队 | 同步进展、暴露阻塞 |
| 周报 + 周会 | 每周一次 | 30-60 分钟 | 全团队 | 周度总结、计划对齐 |
| 迭代评审会 | 每 2 周 | 1-2 小时 | 全团队 + 客户代表 | Demo、反馈收集 |
| 迭代回顾会 | 每 2 周 | 30-60 分钟 | 全开发团队 | 流程改进 |
| 里程碑评审 | 每里程碑 | 2-3 小时 | 全团队 + 管理层 + 客户 | 阶段性决策 |
| 风险审查会 | 每 2 周 | 30 分钟 | PM + 技术负责人 | 风险更新 |
| 变更审查会 | 每 2 周 | 30 分钟 | CCB 成员 | 变更审批 |

### 6.2 每日站会规范

每人回答三个问题：
1. 昨天完成了什么？
2. 今天计划做什么？
3. 有什么阻塞？

**规则**：
- 不超过 15 分钟
- 讨论性问题会后单独沟通
- 站会记录归档到 `docs/会议纪要/站会/`

### 6.3 周报内容模板

```
# 周报 - 第 X 周（日期范围）

## 本周完成
- ...

## 下周计划
- ...

## 风险与阻塞
- ...

## 关键指标
- 计划完成率：XX%
- Bug 数：新增 X / 修复 X / 遗留 X
- 测试覆盖率：XX%
```

### 6.4 迭代评审会流程

```
1. 迭代目标回顾（5 min）
2. 已完成功能 Demo（30-45 min）
3. 未完成工作说明（10 min）
4. 客户/干系人反馈（15 min）
5. 下迭代方向讨论（15 min）
```

### 6.5 里程碑评审会流程

```
1. 里程碑目标达成情况（15 min）
2. 交付物展示与验收（30 min）
3. 风险状况汇报（15 min）
4. 预算与进度偏差分析（15 min）
5. Go/No-Go 决策（15 min）
6. 下阶段计划确认（15 min）
```

### 6.6 沟通工具与渠道

| 沟通类型 | 工具 | 说明 |
|---------|------|------|
| 即时通讯 | 飞书/企业微信/钉钉 | 日常沟通、快速问题 |
| 项目管理 | Jira / 飞书项目 | 任务跟踪、看板管理 |
| 文档协作 | 飞书文档 / Notion / 语雀 | 文档共编、知识库 |
| 代码协作 | GitHub / GitLab | PR Review、Issue 管理 |
| 视频会议 | 飞书会议 / 腾讯会议 | 远程站会、评审会 |
| 邮件 | 企业邮箱 | 正式通知、审批记录 |

### 6.7 沟通纪律
