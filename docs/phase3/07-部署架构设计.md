# 课伴部署架构设计文档

> 版本：v1.1 | 最后更新：2026-07-08

> **v1.1变更说明：** 云端服务新增Supabase本地部署方案（Supabase Studio/Kong/Auth/PostgREST/Realtime），替代独立部署的PostgreSQL和自建Auth服务；服务间通信新增gRPC支持。

## 一、双部署方案总览

课伴采用**前后端分离的双部署架构**：

| 部署单元 | 部署方式 | 说明 |
|---------|---------|------|
| **客户端（PWA）** | CDN静态分发 + Service Worker离线缓存 | 纯前端，无服务端依赖 |
| **云端服务** | Docker + Supabase本地部署 + Kubernetes集群 | 同步服务 + AI服务 + Supabase（Auth/Storage/Realtime/PostgreSQL） |

```
┌─────────────────────────────────────────────────────────────┐
│                    客户端部署（PWA）                           │
│                                                             │
│  用户浏览器 → CDN边缘节点 → 静态资源(JS/CSS/HTML/图片)        │
│                            → Service Worker缓存 → 离线可用   │
└─────────────────────────────────────────────────────────────┘
                            │
                      HTTPS API调用
                            │
┌─────────────────────────────────────────────────────────────┐
│                    云端服务部署（K8s）                         │
│                                                             │
│  Nginx Ingress → Go Sync Service (同步)                     │
│                → Python AI Gateway (AI)                     │
│                → Supabase (Auth/Storage/Realtime/PostgREST)  │
│                                                             │
│  Supabase PostgreSQL · Redis 7 · Supabase Storage              │
└─────────────────────────────────────────────────────────────┘
```

## 二、MVP-1阶段：纯前端PWA部署

### 2.1 部署范围

MVP-1阶段仅需部署前端PWA，**无需后端服务**：

- 所有功能完全本地运行（番茄钟、笔记、闪卡、费曼）
- AI功能显示但不可用（LocalFallback降级）
- 无同步功能（仅本地存储）

### 2.2 CDN部署方案

```
构建产物（dist/）
    ↓
上传到CDN（阿里云OSS/CloudFlare Pages/Vercel）
    ↓
全球边缘节点缓存
    ↓
用户首次访问 → 下载全部静态资源 → Service Worker预缓存
用户再次访问 → Service Worker直接从缓存加载 → 毫秒级启动
```

**CDN选型对比：**

| 平台 | 优势 | 劣势 | 推荐场景 |
|------|------|------|---------|
| **CloudFlare Pages** | 免费额度高、全球CDN、自动HTTPS | 国内访问速度一般 | 海外用户为主 |
| **Vercel** | 部署简单、Preview环境 | 免费额度有限 | 快速迭代 |
| **阿里云OSS + CDN** | 国内访问快、成本可控 | 需手动配置HTTPS/缓存 | 国内用户为主 |
| **GitHub Pages** | 完全免费 | 无法自定义域名(国内)、速度慢 | 演示/Demo |

**推荐：阿里云OSS + CDN（国内）+ CloudFlare Pages（海外备用）**

## 三、MVP-2阶段：K8s集群部署

### 3.1 新增后端服务

MVP-2阶段新增以下云端服务：

| 服务 | 语言/平台 | 端口 | 副本数（初始） | 资源限制 |
|------|----------|------|-------------|----------|
| sync-service | Go | 8080 | 2 | 256MB / 0.5 CPU |
| ai-gateway | Python | 8081 | 2 | 512MB / 1 CPU |
| Supabase Studio | Supabase | 3000 | 1 | 512MB / 0.5 CPU |
| Supabase Kong | Supabase | 8000 | 1 | 256MB / 0.5 CPU |
| Supabase Auth | Supabase | 9999 | 1 | 256MB / 0.5 CPU |
| Supabase PostgREST | Supabase | 3001 | 1 | 256MB / 0.5 CPU |
| Supabase Realtime | Supabase | 4000 | 1 | 512MB / 1 CPU |
| Supabase PostgreSQL | Supabase | 5432 | 1（主从） | 2GB / 2 CPU |
| Redis | - | 6379 | 1（Sentinel） | 512MB / 0.5 CPU |

> v1.1变更：原独立Go Auth Service已替换为Supabase Auth；原MinIO对象存储已替换为Supabase Storage；原独立PostgreSQL已替换为Supabase PostgreSQL。服务间通信新增gRPC支持。

### 3.4 Supabase本地部署架构

> v1.1新增：Supabase提供完整的本地开发环境，通过Docker Compose一键启动所有服务。

#### Supabase本地服务组件

| 组件 | 功能 | 端口 | 说明 |
|------|------|------|------|
| **Supabase Studio** | 管理界面 | 3000 | Web UI管理数据库、Auth、Storage |
| **Kong** | API网关 | 8000 | 统一入口，JWT验证、路由分发 |
| **Auth (GoTrue)** | 认证服务 | 9999 | 注册/登录/Token管理 |
| **PostgREST** | REST API | 3001 | 自动将PostgreSQL表暴露为REST API |
| **Realtime** | 实时订阅 | 4000 | 基于WAL的数据库变更实时推送 |
| **Storage** | 对象存储 | 5000 | 文件上传/下载/访问控制 |
| **PostgreSQL** | 主数据库 | 5432 | 本地PostgreSQL实例 |

#### Docker Compose配置


#### 本地开发工作流

```bash
# 1. 初始化Supabase项目
npx supabase init

# 2. 启动所有本地服务
npx supabase start

# 3. 创建数据库迁移
npx supabase migration new initial_schema

# 4. 应用迁移
npx supabase migration apply

# 5. 生成TypeScript类型
npx supabase gen types typescript --local > src/types/supabase.ts

# 6. 查看Studio管理界面
# 浏览器访问 http://localhost:3000
```

| 特性 | 本地开发环境 | 云端生产环境 |

## 四、环境规划

### 4.1 三环境体系

| 环境 | 用途 | 部署触发 | 数据 | 域名 |
|------|------|---------|------|------|
| **dev** | 开发调试 | push到`develop`分支自动部署 | 测试数据（随机生成） | dev.keban.app |
| **staging** | 预发布验证 | push到`main`分支自动部署 | 生产数据快照（脱敏） | staging.keban.app |
| **prod** | 正式生产 | Git tag发布，手动确认部署 | 真实用户数据 | app.keban.app |

### 4.2 环境隔离策略

```yaml
# K8s Namespace隔离
apiVersion: v1
kind: Namespace
metadata:
  name: keban-dev
---
apiVersion: v1
kind: Namespace
metadata:
  name: keban-staging
---
apiVersion: v1
kind: Namespace
metadata:
  name: keban-prod
```

**每个环境独立的：**
- 数据库实例（dev/staging可共用，prod独立）
- Redis实例
- CDN配置
- 密钥和配置（K8s Secrets / ConfigMaps）

### 4.3 配置管理

```
配置层级：
  代码默认值 → 环境变量 → K8s ConfigMap → K8s Secret（敏感配置）

敏感配置（数据库密码、API密钥、JWT签名密钥）存储在K8s Secrets中
非敏感配置（日志级别、功能开关、超时时间）存储在K8s ConfigMaps中
```

## 五、弹性伸缩策略

### 5.1 水平自动伸缩（HPA）

```yaml
# sync-service HPA
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: sync-service-hpa
  namespace: keban-prod
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: sync-service
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 80
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60    # 扩容稳定窗口60秒
      policies:
        - type: Pods
          value: 2                        # 每次最多扩2个
          periodSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 300   # 缩容稳定窗口5分钟（防抖动）
      policies:
        - type: Pods
          value: 1                        # 每次最多缩1个
          periodSeconds: 120
```

### 5.2 伸缩场景预估

| 场景 | 在线用户 | Sync Pods | AI Pods | DB连接数 |
|------|---------|-----------|---------|---------|
| 日常低峰 | <1000 | 2 | 2 | 20 |
| 学习高峰（晚8-10点） | 5000-10000 | 4-6 | 3-4 | 50 |
| 考试季峰值 | 20000+ | 8-10 | 5-8 | 100 |

### 5.3 数据库伸缩

- **读扩展**：PostgreSQL读写分离，1主2从，读请求路由到从库
- **连接池**：PgBouncer管理连接池，防止连接数暴涨
- **Redis集群**：Redis Sentinel哨兵模式，自动故障转移

## 六、CI/CD流水线设计

### 6.1 流水线总览

```
开发者push代码
    │
    ├─→ [前端流水线] client/
    │     ├─ lint (ESLint + Prettier)
    │     ├─ type-check (TypeScript)
    │     ├─ test (Vitest 单元测试)
    │     ├─ build (Vite 构建)
    │     ├─ lighthouse (性能评分 >90)
    │     └─ deploy (上传CDN)
    │
    └─→ [后端流水线] server/
          ├─ lint (golangci-lint / ruff)
          ├─ test (Go test / pytest)
          ├─ build (Docker镜像构建)
          ├─ scan (Trivy镜像漏洞扫描)
          ├─ push (推送到Container Registry)
          └─ deploy (更新K8s Manifests → ArgoCD同步)
```

### 6.2 GitHub Actions配置

```yaml
# .github/workflows/client-ci.yml
name: Client CI/CD

on:
  push:
    branches: [develop, main]
    paths: ['client/**']
  pull_request:
    branches: [main]
    paths: ['client/**']

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'pnpm'

      - name: Install dependencies
        run: pnpm install --frozen-lockfile
        working-directory: client

      - name: Lint
        run: pnpm lint
        working-directory: client

      - name: Type check
        run: pnpm type-check
        working-directory: client

      - name: Test
        run: pnpm test --coverage
        working-directory: client

      - name: Build
        run: pnpm build
        working-directory: client
        env:
          VITE_API_BASE_URL: ${{ secrets.API_BASE_URL }}

      - name: Deploy to CDN (dev)
        if: github.ref == 'refs/heads/develop'
        run: |
          # 上传到CDN
          echo "Deploying to dev CDN..."

      - name: Deploy to CDN (prod)
        if: github.ref == 'refs/heads/main'
        run: |
          echo "Deploying to prod CDN..."
```

```yaml
# .github/workflows/server-ci.yml
name: Server CI/CD

on:
  push:
    branches: [develop, main]

jobs:

          context: server/sync-service
          push: true

        uses: aquasecurity/trivy-action@master
        with:
          image-ref: registry.keban.app/sync-service:${{ github.sha }}

      - name: Update K8s manifests
        run: |
          # 更新manifests仓库中的镜像标签
          # ArgoCD检测到变更后自动部署
          echo "Updating manifests..."
```

```
代码仓库（keban-app）
    │
    │ CI构建推送镜像

         └─ 健康检查失败 → 自动回滚
```

### 7.2 关键监控指标

| 类别 | 指标 | 告警阈值 | 说明 |
|------|------|---------|------|
| **可用性** | HTTP 5xx错误率 | >1% | 5分钟内超过1%触发告警 |
| **延迟** | API P99延迟 | >500ms | Sync服务P99超过500ms |
| **资源** | CPU使用率 | >80% | 持续5分钟触发扩容 |
| **资源** | 内存使用率 | >85% | 可能内存泄漏 |
| **数据库** | 连接池使用率 | >80% | 接近上限需扩容 |
| **数据库** | 查询延迟P95 | >200ms | 慢查询需优化 |
| **AI服务** | 请求成功率 | <95% | LLM API可能不稳定 |
| **AI服务** | 平均响应时间 | >5s | 模型推理过慢 |
| **客户端** | PWA加载时间 | >3s | CDN或构建产物问题 |
| **客户端** | Service Worker注册失败率 | >5% | 缓存策略可能有bug |

### 7.3 告警通知渠道

```yaml
# AlertManager配置
route:
  receiver: 'default'
  routes:
    - match:
        severity: critical
      receiver: 'pagerduty'     # P0/P1 紧急告警 → PagerDuty
    - match:
        severity: warning
      receiver: 'slack'         # P2/P3 警告 → Slack频道
    - match:
        severity: info
      receiver: 'email'         # 信息级 → 邮件汇总

receivers:
  - name: 'pagerduty'
    pagerduty_configs:
      - service_key: '<pagerduty-key>'

  - name: 'slack'
    slack_configs:
      - api_url: '<slack-webhook>'
        channel: '#keban-alerts'

  - name: 'email'
    email_configs:
      - to: 'team@keban.app'
```

### 7.5 日志规范

```typescript
// 服务端日志格式（JSON结构化日志）
{
  "timestamp": "2026-07-08T10:00:00.000Z",
  "level": "info",            // debug | info | warn | error
  "service": "sync-service",
  "request_id": "req-uuid",   // 请求追踪ID（贯穿全链路）
  "user_id": "user-uuid",
  "method": "POST",
  "path": "/api/sync/batch",
  "status": 200,
  "latency_ms": 45,
  "message": "Sync batch processed",
  "metadata": {
    "operations_count": 12,
    "conflicts": 0
  }
}
```

**日志敏感数据处理：**
- 密码：永远不记录
- Token：仅记录前8位 + `***`
- 邮箱：`u***@example.com`
- 请求体：记录长度，不记录内容
- IP地址：仅记录前两段 `192.168.*.*`
