# 课伴数据同步策略设计文档

> 版本：v1.2 | 最后更新：2026-07-09

> **v1.1变更说明：** 同步引擎从自定义LWW+版本向量升级为Supabase客户端操作，本地PostgreSQL操作直接replay到云端。
>
> **v1.2变更说明 (MVP-2 Alpha 实施)：** 同步策略已 Superseded。实际采用 **HTTP Push/Pull + 操作日志**替代 Supabase 客户端操作 Replay。原因：Replay 模式要求本地与云端共享 PostgreSQL schema，而 Alpha 阶段保留 Dexie.js (IndexedDB)，无法直接 replay。详见 ADR-006 状态更新。

## 一、同步策略对比

| 策略 | 适用场景 | 复杂度 | 冲突处理 | 说明 |
|------|---------|--------|---------|------|
| **Last-Write-Wins (LWW) + 版本向量** | MVP-2初始方案 | ★★☆ | 保留双方版本，用户选择 | 实现简单，覆盖90%单人/少量多设备场景 |
| **CRDT（Yjs/Automerge）** | V2.0升级方案 | ★★★★☆ | 自动无冲突合并 | 适合协同编辑场景，但文档膨胀，学习成本高 |
| **Operational Transform (OT)** | 实时协作场景 | ★★★★★ | 算法保证无冲突 | Google Docs级别方案，MVP阶段过度设计 |
| **冲突保留双方版本（兜底）** | 极端情况兜底 | ★★☆ | 用户手动解决 | 作为LWW和CRDT的共同兜底方案 |

**决策：MVP-2采用Supabase客户端操作 + LWW冲突检测，V2.0可平滑升级至CRDT（Yjs）。**

> v1.1变更：原自定义同步引擎（LWW+版本向量）已升级为Supabase客户端操作。本地PostgreSQL写操作直接replay到云端Supabase，通过数据库级约束和RLS策略保证一致性。

### 2.2 详细步骤说明

**步骤1：本地写操作记录**

所有写操作（create/update/delete）在写入本地PostgreSQL的同时，记录一条操作日志：

<!-- MVP-2 Alpha 修订 (v1.2)：实际写入 Dexie.js (IndexedDB)，通过 IRepository<T> 抽象层隔离实现。操作日志同样存储在 IndexedDB 中。详见 ADR-003 -->

```typescript
async function writeWithLog<T>(
  table: string,
  operation: 'create' | 'update' | 'delete',
  entity: T,
  entityId: string
): Promise<void> {
  const supabase = getLocalSupabase();

  // 在事务内执行写操作+记录日志
  // 1. 执行实际写操作
  if (operation === 'delete') {
    await supabase.from(table).delete().eq('id', entityId);
  } else if (operation === 'create') {
    await supabase.from(table).insert(entity);
  } else {
    await supabase.from(table).update(entity).eq('id', entityId);
  }

  // 2. 记录操作日志
  await supabase.from('operation_logs').insert({
    entity_type: table,
    entity_id: entityId,
    operation,
    data: operation === 'delete' ? null : entity,
    timestamp: new Date().toISOString(),
    synced: false,
    version: await getNextVersion(),
    device_id: getDeviceId(),
  });
}
```

**步骤2：网络恢复时批量推送**

```typescript
async function syncPendingOperations(): Promise<SyncResult> {
  const supabase = getLocalSupabase();

  // 获取所有未同步的操作日志，按时间顺序
  const { data: pendingLogs } = await supabase
    .from('operation_logs')
    .select('*')
    .eq('synced', false)
    .order('timestamp', { ascending: true });

  if (!pendingLogs || pendingLogs.length === 0) {
    return { synced: 0, conflicts: [] };
  }

  // 批量推送到云端Supabase（每批最多100条）

  // <!-- MVP-2 Alpha 修订 (v1.2)：实际推送到 Go sync-service HTTP API，而非云端 Supabase。代码逻辑类似，但目标端点为 /api/v1/sync/push。详见 ADR-006 -->
  const batches = chunk(pendingLogs, 100);
  const result: SyncResult = { synced: 0, conflicts: [] };

  for (const batch of batches) {
    const response = await fetch('/api/v1/sync/push', {
      method: 'POST',
      headers: { 'Authorization': `Bearer ${getToken()}` },
      body: JSON.stringify({
        device_id: getDeviceId(),
        operations: batch,
      }),
    });

    const batchResult = await response.json();
    result.synced += batchResult.synced;
    result.conflicts.push(...batchResult.conflicts);

    // 标记已同步的操作
    const syncedIds = batchResult.synced_ids ?? [];
    if (syncedIds.length > 0) {
      await supabase
        .from('operation_logs')
        .update({ synced: true })
        .in('id', syncedIds);
    }
  }

  return result;
}
```

## 四、版本向量实现方案

### 4.1 版本向量数据结构

```typescript
// 版本向量：记录每个设备对某实体的最新版本号
interface VersionVector {
  [deviceId: string]: number;  // deviceId → 该设备对该实体的最新操作版本号
}

// 示例：某条笔记的版本向量
// { "device-laptop-001": 42, "device-phone-002": 17 }
// 表示：laptop对该笔记最后操作版本为42，phone为17
```

### 4.2 版本向量比较规则

```typescript
type VectorRelation = 'equal' | 'ancestor' | 'descendant' | 'concurrent';

function compareVectors(a: VersionVector, b: VersionVector): VectorRelation {
  const allDevices = new Set([...Object.keys(a), ...Object.keys(b)]);

  let aGreater = false;
  let bGreater = false;

  for (const device of allDevices) {
    const aVersion = a[device] ?? 0;
    const bVersion = b[device] ?? 0;

    if (aVersion > bVersion) aGreater = true;
    if (bVersion > aVersion) bGreater = true;
  }

  if (!aGreater && !bGreater) return 'equal';        // 完全相同
  if (aGreater && !bGreater) return 'descendant';    // a是b的后代（a更新）
  if (!aGreater && bGreater) return 'ancestor';      // a是b的祖先（b更新）
  return 'concurrent';                                 // 并发（真正冲突）
}
```

### 4.3 冲突检测流程

```
客户端A推送操作: { entityId: "note-001", version: 43, deviceId: "A" }
客户端B推送操作: { entityId: "note-001", version: 18, deviceId: "B" }

服务端处理：
1. 查询 note-001 当前版本向量: { "A": 42, "B": 17 }
2. 处理A的操作：42→43，关系=descendant（正常更新）
   更新版本向量为: { "A": 43, "B": 17 }
3. 处理B的操作：17→18，但A已更新到43
   compareVectors({"B": 18}, {"A": 43, "B": 17}) = 'concurrent'
   → 检测到冲突！保留双方版本，标记冲突
```

## 六、弱网/断网/重连处理策略

### 6.1 网络状态管理

```typescript
class NetworkManager {
  private status: 'online' | 'offline' | 'weak' = 'online';
  private listeners: Set<(status: NetworkStatus) => void> = new Set();

  constructor() {
    // 监听浏览器原生网络事件
    window.addEventListener('online', () => this.setStatus('online'));
    window.addEventListener('offline', () => this.setStatus('offline'));

    // 心跳检测（弱网识别）
    this.startHeartbeat();
  }

  private async startHeartbeat(): Promise<void> {
    setInterval(async () => {
      if (this.status === 'offline') return;

      const start = performance.now();
      try {
        await fetch('/api/health', { method: 'HEAD', cache: 'no-store' });
        const latency = performance.now() - start;
        this.setStatus(latency > 3000 ? 'weak' : 'online');
      } catch {
        this.setStatus('offline');
      }
    }, 10000); // 每10秒检测一次
  }

  private setStatus(status: NetworkStatus): void {
    if (this.status !== status) {
      this.status = status;
      this.listeners.forEach(fn => fn(status));
    }
  }

  onStatusChange(fn: (status: NetworkStatus) => void): () => void {
    this.listeners.add(fn);
    return () => this.listeners.delete(fn);
  }
}
```

### 6.3 重连同步策略

```typescript
// 指数退避重试
class RetryStrategy {
  private maxRetries = 5;
  private baseDelay = 1000;  // 1秒
  private maxDelay = 60000;  // 60秒

  async execute<T>(fn: () => Promise<T>): Promise<T> {
    for (let attempt = 0; attempt < this.maxRetries; attempt++) {
      try {
        return await fn();
      } catch (error) {
        if (attempt === this.maxRetries - 1) throw error;
        const delay = Math.min(
          this.baseDelay * Math.pow(2, attempt) + Math.random() * 1000,
          this.maxDelay
        );
        await sleep(delay);
      }
    }
    throw new Error('Max retries exceeded');
  }
}
```

## 七、离线操作队列与重放机制

### 7.1 队列设计

```typescript
interface OfflineQueueItem {
  id: string;           // 队列项ID
  operationLogId: string; // 关联的操作日志ID
  retryCount: number;    // 重试次数
  lastError?: string;    // 最近错误信息
  status: 'pending' | 'retrying' | 'failed' | 'conflict';
  createdAt: number;
}

class OfflineQueue {
  // 操作日志已在IndexedDB中append-only记录
  // 离线队列仅追踪同步状态

  async enqueue(operationLogId: string): Promise<void> {
    await db.offlineQueue.add({
      id: crypto.randomUUID(),
      operationLogId,
      retryCount: 0,
      status: 'pending',
      createdAt: Date.now(),
    });
  }

  async replay(): Promise<ReplayResult> {
    const pending = await db.offlineQueue
      .where('status').anyOf(['pending', 'retrying'])
      .sortBy('createdAt');

    const result: ReplayResult = { success: 0, failed: 0, conflicts: 0 };

    for (const item of pending) {
      try {
        const log = await db.operationLogs.get(item.operationLogId);
        if (!log) continue;

        await retryStrategy.execute(() => pushToServer(log));

        await db.offlineQueue.update(item.id, { status: 'success' });
        await db.operationLogs.update(item.operationLogId, { synced: true });
        result.success++;
      } catch (error: any) {
        const retryCount = item.retryCount + 1;
        if (retryCount >= MAX_RETRIES) {
          await db.offlineQueue.update(item.id, {
            status: 'failed',
            retryCount,
            lastError: error.message,
          });
          result.failed++;
        } else {
          await db.offlineQueue.update(item.id, {
            status: 'retrying',
            retryCount,
            lastError: error.message,
          });
          result.failed++;
        }
      }
    }

    return result;
  }
}
```

### 7.2 重放顺序保证

操作日志按 `timestamp` 升序重放，确保操作顺序与用户实际操作顺序一致。对于同一实体的多次更新，仅最后一次状态需要成功推送，中间的update操作若失败可安全跳过。

### 8.4 数据格式兼容性

```typescript
// 旧操作日志 → Yjs操作 转换器
function migrateOperationToYjs(
  log: OperationLog,
  ydoc: Y.Doc
): void {
  const ymap = ydoc.getMap(log.entityType);

  switch (log.operation) {
    case 'create':
    case 'update': {
      const data = JSON.parse(log.data!);
      const yentity = ymap.get(log.entityId) ?? new Y.Map();
      for (const [key, value] of Object.entries(data)) {
        yentity.set(key, value);
      }
      ymap.set(log.entityId, yentity);
      break;
    }
    case 'delete':
      ymap.delete(log.entityId);
      break;
  }
}
```

### 8.5 升级风险控制

| 风险 | 缓解措施 |
|------|---------|
| 数据迁移丢失 | 迁移前完整备份，迁移后校验checksum |
| 新旧版本不兼容 | 双写兼容期至少4周，灰度发布 |
| CRDT文档膨胀 | 定期压缩Yjs文档历史（gc） |
| 性能下降 | 大文档拆分，懒加载未访问文档 |
