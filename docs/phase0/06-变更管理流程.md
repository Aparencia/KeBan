## 一、变更管理原则

### 1.1 核心原则

| 编号 | 原则 | 说明 |
|------|------|------|
| P-1 | **M1后严格执行变更控制** | 需求冻结里程碑（M1，第4周）后，所有需求变更必须走正式变更流程 |
| P-2 | **MVP-1范围锁定** | MVP-1范围一经锁定，不可追加新功能，确保第28周按时交付 |
| P-3 | **新需求进入MVP-2 Backlog** | 所有MVP-1范围外的新需求，统一纳入MVP-2 Backlog进行优先级排序 |
| P-4 | **变更影响必须量化** | 每个变更申请必须明确评估对范围、时间、成本、风险四个维度的影响 |
| P-5 | **变更预算有限** | 总工作量预留10-15%作为变更缓冲（约5-8人月），超出需追加预算审批 |
| P-6 | **变更决策透明** | 所有变更决策（批准/拒绝/延期）记录在案，全员可查 |

### 1.2 变更管控分级

```
M1之前（W01-W04）：需求调整期
  └── 需求仍在细化中，调整相对灵活
  └── 但重大方向性变更仍需PM和技术负责人确认

M1之后 → MVP-1发布（W05-W28）：严格冻结期
  └── 仅接受"微小变更"（bug修复、体验优化）
  └── 新增功能一律拒绝，进入MVP-2 Backlog
  └── 重大变更需CCB审批 + 项目发起人签字

MVP-1之后 → MVP-2发布（W29-W36）：受控开放期
  └── MVP-2范围内的需求变更走标准流程
  └── MVP-2范围外的需求进入正式上线Backlog

Beta公测之后（W44+）：紧急修复期
  └── 仅接受阻塞性Bug修复和关键体验问题
  └── 功能变更一律推迟至后续版本
```

## 二、变更流程（五步）

### 流程总览

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  第一步   │    │  第二步   │    │  第三步   │    │  第四步   │    │  第五步   │
│ 提交变更  │───→│ 影响评估  │───→│ CCB审批  │───→│ 实施变更  │───→│ 验证确认  │
│  申请     │    │          │    │          │    │          │    │          │
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
   申请人          产品经理         CCB委员会       开发团队        测试+客户
                 + 技术负责人                                    + 申请人
```

---

### 第一步：提交变更申请

**触发条件**：任何项目干系人（客户、产品经理、开发团队）认为需要修改已确定的需求。

**操作要求**：
1. 填写《变更申请表》（见第四章模板）
2. 清晰描述变更内容和变更原因
3. 注明变更的紧急程度（紧急/一般/低优先）
4. 提交至项目经理登记

**时间要求**：
- 变更申请随时可提交
- 提交后1个工作日内进入影响评估环节

---

### 第二步：影响评估

**责任人**：产品经理 + 技术负责人

**评估维度**：

| 维度 | 评估内容 | 量化标准 |
|------|----------|----------|
| **范围影响** | 是否新增/删除/修改功能，是否影响其他模块 | 影响功能点数 |
| **时间影响** | 是否导致里程碑延迟，延迟多少天 | 延迟天数 |
| **成本影响** | 需要额外多少工作量，是否超出变更预算 | 增加人天数 |
| **风险影响** | 是否引入新的技术风险或业务风险 | 新增风险数量及等级 |
| **质量影响** | 是否影响已确定的质量标准 | 测试覆盖率变化 |

**评估输出**：
- 完成《变更申请表》中的"影响评估"部分
- 给出明确的评估结论和建议（接受/拒绝/延期）
- 3个工作日内完成评估

---

### 第三步：CCB审批

**CCB（变更控制委员会）组成**：

| 角色 | 职责 |
|------|------|
| 项目经理（主席） | 主持会议，综合评估，最终决策 |
| 产品经理 | 评估需求价值和优先级 |
| 技术负责人 | 评估技术可行性和工作量 |
| 客户代表（如适用） | 确认业务价值和验收标准 |

**审批决策类型**：

| 决策 | 说明 | 后续动作 |
|------|------|----------|
| **批准** | 变更纳入项目范围 | 进入第四步：实施变更 |
| **有条件批准** | 批准但需调整实施方案 | 修改后重新评估 |
| **拒绝** | 变更不纳入当前版本 | 记录拒绝原因，转入Backlog |
| **延期** | 当前不适合实施，后续版本再评估 | 纳入下一版本Backlog |

**审批规则**：

| 变更类型 | 审批流程 | 决策时限 |
|----------|----------|----------|
| 微小变更（工作量≤2人天） | PM直接审批 | 1个工作日 |
| 一般变更（工作量3-10人天） | CCB会议审批 | 3个工作日 |
| 重大变更（工作量>10人天或影响里程碑） | CCB + 项目发起人审批 | 5个工作日 |

---

### 第四步：实施变更

**操作要求**：
1. 更新需求文档，标注变更内容（含变更编号）
2. 调整项目计划（Sprint Backlog、甘特图）
3. 更新受影响的架构设计/技术方案（如有）
4. 更新测试用例和测试计划
5. 通知所有相关人员变更内容
6. 执行开发和测试

**文档更新检查清单**：

- [ ] 需求文档更新
- [ ] 项目计划/Sprint Backlog更新
- [ ] 技术设计文档更新（如有）
- [ ] 测试用例更新
- [ ] 变更追踪表更新
- [ ] 变更预算消耗更新

---

### 第五步：验证确认

**操作要求**：
1. 测试工程师按更新后的测试用例执行验证测试
2. 确认变更实现符合变更申请中的需求描述
3. 确认变更未引入回归问题
4. 申请人/客户确认变更结果
5. 关闭变更单，更新变更追踪表状态为"已完成"

**验证通过标准**：
- 变更功能实现完整，符合需求描述
- 无新增阻塞性或严重Bug
- 回归测试通过
- 申请人签字确认

## 三、变更分类

### 3.1 微小变更（可直接纳入）

**定义**：不影响项目范围、时间、成本的小幅调整。

| 类型 | 示例 | 处理方式 |
|------|------|----------|
| Bug修复 | 已确认的功能Bug | 直接纳入当前Sprint |
| UI微调 | 按钮位置/颜色/字号调整 | 直接纳入当前Sprint |
| 文案修正 | 提示文案、错误信息修正 | 直接纳入当前Sprint |
| 性能微调 | 局部性能优化（<0.5人天） | 直接纳入当前Sprint |

**审批要求**：PM审批即可，无需CCB会议

---

### 3.2 一般变更（需标准审批）

**定义**：影响工作量但不影响里程碑的变更。

| 类型 | 示例 | 处理方式 |
|------|------|----------|
| 功能调整 | 修改已有功能的交互方式 | CCB审批 |
| 新增子功能 | 在已有模块中新增小功能点 | CCB审批 |
| 技术方案变更 | 更换某个组件的实现方案 | CCB审批 |
| 测试策略调整 | 增加/减少测试覆盖范围 | CCB审批 |

**审批要求**：CCB会议审批，3个工作日决策

---

### 3.3 重大变更（需正式审批+费用调整）

**定义**：影响里程碑、显著增加工作量或改变产品方向的变更。

| 类型 | 示例 | 处理方式 |
|------|------|----------|
| 新增模块 | 增加全新的功能模块 | CCB + 项目发起人审批 |
| 范围扩大 | MVP-1追加非核心功能 | CCB + 项目发起人审批（通常拒绝） |
| 技术重构 | 更换核心框架或存储方案 | CCB + 项目发起人审批 |
| 架构变更 | 改变双模式架构设计 | CCB + 项目发起人审批 |
| 工期调整 | 延长或压缩里程碑 | CCB + 项目发起人审批 |

**审批要求**：CCB + 项目发起人双重审批，5个工作日决策，需重新评估预算

## 六、变更预算

### 6.1 变更缓冲池

| 项目 | 数值 |
|------|------|
| **总工作量** | 38-50人月 |
| **变更缓冲比例** | 10-15% |
| **变更缓冲总量** | 约5-8人月 |
| **MVP-1阶段缓冲** | 约2-3人月 |
| **MVP-2阶段缓冲** | 约2-3人月 |
| **正式上线阶段缓冲** | 约1-2人月 |

### 6.2 变更预算消耗规则

| 规则 | 说明 |
|------|------|
| **微小变更不消耗预算** | Bug修复、UI微调等日常工作不计入变更预算 |
| **一般变更按评估消耗** | 按影响评估的工作量从缓冲池中扣除 |
| **重大变更需追加审批** | 若单次变更超过缓冲池剩余量的30%，需项目发起人额外审批 |
| **缓冲池耗尽预警** | 缓冲消耗超过80%时，自动触发PM和发起人审视 |
| **缓冲池耗尽** | 后续变更需追加预算审批，或从后续版本中裁剪功能 |

### 6.3 变更预算追踪

| 阶段 | 缓冲总量 | 已消耗 | 剩余 | 消耗比例 | 状态 |
|------|----------|--------|------|----------|------|
| MVP-1 | 2-3人月 | 0 | 2-3 | 0% | ✅ 正常 |
| MVP-2 | 2-3人月 | 0 | 2-3 | 0% | ✅ 正常 |
| 正式上线 | 1-2人月 | 0 | 1-2 | 0% | ✅ 正常 |
| **合计** | **5-8人月** | **0** | **5-8** | **0%** | ✅ 正常 |

## 七、变更管理纪律

### 7.1 红线规则（不可违反）

| 红线 | 说明 | 违反后果 |
|------|------|----------|
| **禁止口头变更** | 所有变更必须书面申请，不接受口头约定的需求变更 | 开发团队有权拒绝执行 |
| **禁止跳过评估** | 任何变更（包括微小变更）都必须经过影响评估 | 变更无效，需重新走流程 |
| **禁止超预算执行** | 变更工作量不得超过审批确认的工作量 | 超出部分需重新审批 |
| **MVP-1冻结后禁止加功能** | MVP-1范围锁定后，任何新增功能一律转入MVP-2 | 无例外 |

### 7.2 常见变更陷阱

| 陷阱 | 描述 | 应对策略 |
|------|------|----------|
| **"就加一个小功能"** | 每个"小功能"都看似无害，累积起来就是范围蔓延 | 严格执行变更流程，量化累计影响 |
| **"客户说必须要"** | 用客户压力推动变更 | 引导客户将需求进入MVP-2 Backlog |
| **"技术上很简单"** | 技术简单≠没有影响（测试、文档、集成等） | 全面评估，不仅看开发工作量 |
| **"先做了再说"** | 跳过流程先开发，后补变更单 | 严禁，开发团队有权拒绝 |

## 八、流程改进

### 8.1 回顾机制

- 每个里程碑节点回顾变更管理执行情况
- 分析变更数量、变更原因分布、流程执行合规度
- 根据回顾结果优化变更管理流程

### 8.2 度量指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 变更申请总数 | 越少越好（M1后） | 反映需求质量 |
| 变更批准率 | 60-80% | 过高说明需求不清晰，过低说明变更控制过严 |
| 平均变更处理时长 | <5个工作日 | 从提交到审批的平均时间 |
| 变更预算消耗率 | <80% | 保持健康余量 |
| 变更导致的工期延迟 | 0天（MVP-1） | MVP-1不应因变更延迟 |
