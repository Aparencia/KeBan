## 一、风险评估标准说明

### 1.1 影响等级定义

| 等级 | 定义 | 示例 |
|------|------|------|
| **极高** | 项目可能被迫终止或造成重大经济损失 | 核心数据不可恢复丢失 |
| **高** | 关键里程碑延迟超过2周，或核心功能无法交付 | 主存储方案不可用 |
| **中** | 部分功能延迟或质量降级，但不影响核心交付 | 特定浏览器兼容性降级 |
| **低** | 对项目影响轻微，可轻松吸收 | 非核心功能小bug |

### 1.2 发生概率定义

| 等级 | 概率区间 | 描述 |
|------|----------|------|
| **高** | >70% | 很可能发生，已有前兆 |
| **中高** | 50-70% | 有较大可能发生 |
| **中** | 30-50% | 有一定可能发生 |
| **低** | 10-30% | 发生可能性较小 |
| **极低** | <10% | 几乎不会发生 |

### 1.3 风险值计算

风险值 = 影响等级 × 发生概率

| | 低概率 | 中概率 | 中高概率 | 高概率 |
|---|---|---|---|---|
| **低影响** | 1 | 2 | 3 | 4 |
| **中影响** | 2 | 4 | 6 | 8 |
| **高影响** | 3 | 6 | 9 | 12 |
| **极高影响** | 4 | 8 | 12 | 16 |

风险等级划分：1-3（低）、4-6（中）、8-9（高）、12-16（极高）

## 三、风险详细分析

### NR-1：浏览器存储容量不足（50MB-2GB限制）

| 属性 | 内容 |
|------|------|
| **风险类别** | 技术风险 |
| **影响等级** | 高 |
| **发生概率** | 中 |
| **风险值** | 6（中） |
| **风险描述** | 各浏览器对IndexedDB的存储配额不同（Chrome约为磁盘空间的60%，Firefox约50%，Safari有1GB硬限制），当用户笔记和附件数据量增长时，可能触及存储上限 |
| **潜在影响** | 用户无法继续创建笔记，应用功能受限，用户体验严重受损 |
| **应对策略** | **缓解**：① 实现存储空间监控和预警机制（使用量>80%时提醒）；② 提供数据导出功能，支持导出为本地JSON文件；③ 实现附件压缩（图片自动压缩、文档懒加载）；④ 云同步作为"无限存储"的增值方案 |
| **责任人** | 客户端架构师 |
| **状态** | 待处理 |
| **触发条件** | PoC验证1万笔记+5万闪卡场景下存储超过500MB |

### NR-2：同步冲突导致数据丢失

| 属性 | 内容 |
|------|------|
| **风险类别** | 技术风险 |
| **影响等级** | 高 |
| **发生概率** | 中高 |
| **风险值** | 9（高） |
| **风险描述** | 用户多设备同时编辑同一条笔记或闪卡，同步时产生冲突，若冲突解决策略不当可能导致数据丢失或覆盖 |
| **潜在影响** | 用户辛苦编辑的笔记内容丢失，造成极大负面体验，严重损害产品信任度 |
| **应对策略** | **缓解**：① MVP-2采用CRDT（如Yjs或Automerge）实现无冲突自动合并；② 冲突解决策略优先保留"最晚编辑版本"并保留历史版本；③ 提供冲突检测和人工解决界面；④ 所有数据变更保留操作日志（Undo History） |
| **责任人** | 离线同步专家 |
| **状态** | 待处理 |
| **触发条件** | 多设备同步测试中出现数据覆盖现象 |

### NR-3：双模式代码维护成本翻倍

| 属性 | 内容 |
|------|------|
| **风险类别** | 架构风险 |
| **影响等级** | 高 |
| **发生概率** | 高 |
| **风险值** | 12（极高） |
| **风险描述** | 本地模式和AI增强模式双轨并行，每个功能模块都需要维护本地版和云端版两套逻辑，代码复杂度和维护成本可能翻倍 |
| **潜在影响** | 开发效率降低，bug率上升，团队精力分散，工期延长 |
| **应对策略** | **规避**：① 采用策略模式+依赖注入，本地/AI版本通过统一接口切换，而非硬编码两套逻辑；② AI功能通过独立插件包发布，核心模块完全不感知AI存在；③ 建立严格的模块边界和接口契约；④ Code Review重点关注模式分支复杂度 |
| **责任人** | 技术负责人/架构师 |
| **状态** | 待处理 |
| **触发条件** | 单个功能模块中AI相关代码占比超过30% |

### NR-4：离线/在线频繁切换体验割裂

| 属性 | 内容 |
|------|------|
| **风险类别** | 体验风险 |
| **影响等级** | 中 |
| **发生概率** | 中高 |

| **潜在影响** | 用户对产品稳定性产生质疑，影响留存率和口碑 |

| **责任人** | 前端工程师（UX方向） |
| **状态** | 待处理 |
| **触发条件** | 用户测试中50%以上用户反馈"切换体验不佳" |

## 四、Top 3 需立即行动的风险

### 1. NR-2：同步冲突导致数据丢失（风险值：9，高）

**紧急原因**：数据是用户最核心的资产，同步冲突可能导致不可逆的数据丢失，是产品信任度的根基问题。

**立即行动项**：
- [ ] W02：启动CRDT技术选型调研（Yjs vs Automerge）
- [ ] W03：完成同步冲突场景PoC，验证自动合并效果
- [ ] W04：制定同步冲突解决策略文档，明确"自动合并优先 + 人工解决兜底"方案

---

### 2. NR-3：双模式代码维护成本翻倍（风险值：12，极高）

**紧急原因**：双模式是课伴的核心架构特色，若架构设计不当，将从根本上拖垮开发效率和代码质量。

**立即行动项**：
- [ ] W01：架构师完成"策略模式+插件化"的架构设计方案
- [ ] W02：建立模块边界规范和接口契约模板
- [ ] W03：完成一个模块（建议番茄钟）的双模式架构示范实现

---

### 3. NR-8：工期延长导致市场窗口错失（风险值：6，中）

**紧急原因**：2026年9月开学季是黄金市场窗口，错过将导致获客成本大幅上升，且需等待至2027年2月春季学期。

**立即行动项**：
- [ ] W01：制定严格的MVP-1范围清单（含明确的"不包含"清单）
- [ ] W01：建立双周Sprint机制和进度追踪看板
- [ ] W04：第一次进度审视点，评估是否需要调整范围

## 六、风险审查机制

### 6.1 审查周期

| 审查类型 | 频率 | 参与者 | 输出物 |
|----------|------|--------|--------|
| **日常监控** | 每日 | 各风险责任人 | 风险状态更新（如有变化） |
| **Sprint评审** | 每两周 | 全团队 | 风险登记册更新版 |
| **里程碑审查** | 每个里程碑节点 | 项目经理 + 架构师 + PM | 风险评估报告 |
| **紧急审查** | 触发式 | 相关人员 | 风险应急处理记录 |

### 6.2 审查流程

1. 风险责任人汇报各自风险状态变化
2. 评估是否有新增风险需要登记
3. 更新风险值（影响等级和概率可能变化）
4. 审视Top 3风险行动项完成情况
5. 决定是否需要启动新的缓解/规避措施
6. 更新风险登记册版本

### 6.3 风险升级机制

| 风险等级 | 报告层级 | 处理时效 |
|----------|----------|----------|
| 极高（12-16） | 项目发起人 + 全体管理层 | 24小时内响应 |
| 高（8-9） | 项目经理 + 技术负责人 | 3个工作日内响应 |
| 中（4-6） | 风险责任人 + 项目经理 | Sprint内响应 |
| 低（1-3） | 风险责任人 | 下次审查时处理 |
