## 概述

本报告对课伴（KeBan）项目的核心技术方案进行系统性可行性评估，涵盖本地存储方案、SM-2间隔重复算法、AI可选接入、富文本编辑器、Service Worker离线方案五大技术方向。每个方向从成熟度、性能、兼容性、风险等维度进行量化评估，最终给出综合评估结论。

## 一、本地存储方案成熟度评估

### 1.1 方案对比矩阵

| 评估维度 | IndexedDB (via Dexie.js) | SQLite WASM (wa-sqlite) | OPFS | localStorage |
|----------|--------------------------|------------------------|------|-------------|
| **成熟度** | ★★★★★ 10+年，W3C标准 | ★★★☆☆ 较新，社区维护 | ★★☆☆☆ 实验性API | ★★★★★ 最成熟 |
| **存储容量** | 50MB ~ 2GB（浏览器决定） | 理论无上限（受限于WASM内存） | 理论无上限 | 5-10MB |
| **性能** | 异步，中高性能 | 同步/异步，高性能 | 高性能，低延迟 | 同步，小数据快 |
| **数据类型** | 结构化对象，支持索引 | 完整SQL支持 | 文件/二进制数据 | 字符串KV |
| **事务支持** | 完整事务 | 完整ACID事务 | 无事务 | 无 |
| **Chrome兼容** | ✅ 完美 | ✅ 完美 | ✅ 支持 | ✅ 完美 |
| **Firefox兼容** | ✅ 完美 | ✅ 完美 | ⚠️ 部分支持 | ✅ 完美 |
| **Safari兼容** | ⚠️ 有已知Bug | ⚠️ WASM限制 | ❌ 不支持 | ✅ 完美 |
| **社区生态** | Dexie.js 60k+周下载 | wa-sqlite 活跃开发 | 早期阶段 | 原生API |
| **学习曲线** | 低 | 中 | 低 | 极低 |

### 1.2 详细分析

#### IndexedDB + Dexie.js（推荐方案）

**优势：**
- Dexie.js提供优雅的Promise-based API，大幅简化IndexedDB原生API的复杂性
- 支持实时查询（liveQuery）、索引优化、批量操作
- Dexie.js Cloud可作为同步层，降低后期云同步开发成本
- 生态成熟，大量生产级应用验证（Notion早期版本、Trello等）

**风险：**
- Safari对IndexedDB有已知Bug（Private Mode下容量限制为0、某些版本不支持持久化）
- 大容量数据（>500MB笔记/附件）场景下性能需额外优化
- 浏览器清除数据操作将导致全部本地数据丢失

**缓解措施：**
- 实现定期本地导出功能，用户可手动备份为JSON文件
- 对Safari用户检测存储可用性，降级提示
- 数据压缩存储，减少空间占用

#### SQLite WASM (wa-sqlite)（备选方案）

**优势：**
- 完整的SQL支持，复杂查询能力强
- 数据以文件形式存在OPFS中，便于备份导出
- SQLite有数十亿设备的部署验证

**风险：**
- WASM包体积较大（约1MB gzip），影响首屏加载
- 与JavaScript对象之间的序列化开销
- Safari对WASM的支持不如Chrome/Firefox完善
- 社区相对较小，遇到问题排查成本高

### 1.3 评估结论

| 评估项 | 结论 |
|--------|------|
| **推荐方案** | IndexedDB + Dexie.js |
| **可行性评级** | **高** ✅ |
| **信心度** | 90% |
| **关键风险** | Safari兼容性、浏览器清除数据导致丢失 |
| **下一步** | 搭建PoC验证大数据量场景性能表现 |

## 二、SM-2间隔重复算法评估

### 2.1 算法概述

SM-2（SuperMemo 2）算法由Piotr Wozniak于1987年提出，至今已有30+年历史，是最经典、最广泛使用的间隔重复算法。

**核心原理：**
- 根据用户对每张卡片的回忆质量（0-5分）动态调整下次复习间隔
- 记忆难度（EF, Easiness Factor）随复习表现动态调整
- 复习间隔呈指数增长：1天 → 6天 → 约15天 → 约35天...

### 2.2 业界验证

| 验证维度 | 说明 |
|----------|------|
| **历史验证** | 30+年学术研究和实践验证，论文引用数千篇 |
| **产品验证** | Anki（全球数千万用户）、SuperMemo、Quizlet均基于SM-2或其变体 |
| **效果验证** | 大量认知科学研究证实间隔重复比集中复习效率提升50-100% |
| **算法变体** | SM-17、FSRS等新算法在SM-2基础上优化，但SM-2仍是"够用且简单"的首选 |

### 2.3 端侧实现评估

| 评估维度 | 评估结论 |
|----------|----------|
| **JavaScript实现** | 完全可行，核心算法约200行代码 |
| **计算性能** | 极低开销，单张卡片复习调度<1ms |
| **存储需求** | 每张卡片仅需存储EF、间隔、下次复习日期等少量元数据 |
| **离线运行** | 完全离线，算法本身不需要任何网络请求 |
| **算法升级路径** | 预留接口，后期可升级为FSRS等更智能的算法 |

### 2.4 评估结论

| 评估项 | 结论 |
|--------|------|
| **可行性评级** | **高** ✅ |
| **信心度** | 98% |
| **关键风险** | 无明显技术风险 |
| **下一步** | 实现核心算法模块，编写单元测试覆盖各评分场景 |

## 三、AI可选接入评估

### 3.1 AI能力矩阵

| AI能力 | 技术方案 | API成熟度 | 延迟 | 成本估算 |
|--------|----------|-----------|------|----------|
| **NLP笔记摘要** | GPT-4o / Claude 3.5 / Qwen-Max | ★★★★★ 生产级 | 2-5秒 | ~¥0.03/次 |
| **智能闪卡生成** | GPT-4o-mini / Qwen-Turbo | ★★★★★ 生产级 | 3-8秒 | ~¥0.01/次 |
| **费曼AI评估** | GPT-4o / Claude 3.5 | ★★★★☆ 稳定 | 5-15秒 | ~¥0.05/次 |
| **知识点图谱** | 自建 + LLM辅助 | ★★★☆☆ 需定制 | N/A | 开发成本高 |
| **语音转文字** | Whisper API / 本地Whisper | ★★★★☆ 稳定 | 实时~3秒 | ~¥0.01/分钟 |

### 3.2 架构设计

采用**插件化架构**，AI能力作为可选插件接入：

```
┌─────────────────────────────────────────┐
│              应用核心层                    │
│  （番茄钟 / 笔记 / 闪卡 / 费曼）          │
├─────────────────────────────────────────┤
│            AI接口抽象层                    │
│  （统一API接口，屏蔽底层模型差异）          │
├────────┬────────┬────────────────────────┤
│ OpenAI │ Claude │ Qwen │ 本地模型(未来)   │
│ Adapter│ Adapter│ Adapter│  Adapter       │
└────────┴────────┴────────────────────────┘
```

**关键设计原则：**
- AI接口层与核心业务完全解耦
- 各模块调用AI通过统一接口，不关心底层实现
- 支持动态切换AI供应商，避免供应商锁定
- 本地模式完全不加载AI相关代码

### 3.3 风险评估

| 风险项 | 影响 | 概率 | 缓解措施 |
|--------|------|------|----------|
| API服务不稳定 | 高 | 中 | 多供应商备选、降级提示、重试机制 |
| API成本失控 | 中 | 中 | 使用量限制、缓存策略、小模型替代 |
| AI输出质量不稳定 | 中 | 中高 | Prompt工程优化、人工审核、用户反馈机制 |
| 国内网络访问限制 | 高 | 中 | 优先对接国内模型（Qwen）、自建代理网关 |
| 模型版本升级导致行为变化 | 中 | 中 | 锁定模型版本、充分测试后再升级 |

### 3.4 评估结论

| 评估项 | 结论 |
|--------|------|
| **可行性评级** | **中高** ⚠️ |
| **信心度** | 75% |
| **关键风险** | 网络依赖、服务稳定性、成本控制 |
| **缓解策略** | 插件化架构，AI功能不影响核心功能可用性 |
| **下一步** | 完成3家AI供应商的PoC对比测试 |

## 四、富文本编辑器评估

### 4.1 方案对比

| 评估维度 | ProseMirror | TipTap | Slate |
|----------|-------------|--------|-------|
| **架构** | 底层框架 | 基于ProseMirror的封装 | 基于React的框架 |
| **可扩展性** | ★★★★★ 极强 | ★★★★★ 极强 | ★★★★☆ 强 |
| **学习曲线** | 陡峭 | 平缓 | 中等 |
| **文档质量** | 中等 | 优秀 | 中等 |
| **社区活跃度** | 中 | 高（GitHub 28k+ stars） | 中（活跃度下降） |
| **框架无关性** | ✅ 完全无关 | ✅ 提供Vue/React封装 | ❌ React绑定 |
| **协同编辑** | 原生支持 | 支持（HocusPocus） | 有限支持 |
| **自定义节点** | 强大但复杂 | 简洁且强大 | 强大 |
| **模板渲染** | 需自行实现 | 支持良好 | 需自行实现 |
| **离线支持** | 天然支持 | 天然支持 | 天然支持 |
| **性能** | 优秀 | 优秀 | 良好 |
| **维护状态** | 活跃 | 非常活跃 | 活跃度下降 |

### 4.2 TipTap 详细评估

**选择TipTap的核心理由：**

1. **基于ProseMirror**：继承ProseMirror强大的文档模型和插件系统，同时大幅降低学习曲线
2. **框架无关**：项目当前技术栈为React，但TipTap不绑定特定框架，未来技术迁移无阻碍
3. **扩展系统完善**：内置40+扩展（Heading、List、Table、CodeBlock等），满足笔记模块所有需求
4. **自定义节点**：支持自定义节点类型，可实现课伴特有的"知识卡片嵌入"、"闪卡关联标记"等定制功能
5. **AI集成友好**：支持在编辑器中嵌入AI生成的内容（如摘要、建议），有良好的内容插入API
6. **离线能力**：编辑器本身纯前端运行，与本地存储方案完美配合

**风险：**
- TipTap Cloud（付费协同方案）成本较高，但MVP-1不涉及协同，暂不影响
- 深度自定义需要理解底层ProseMirror，仍有一定学习成本

### 4.3 评估结论

| 评估项 | 结论 |
|--------|------|
| **推荐方案** | TipTap |
| **可行性评级** | **高** ✅ |
| **信心度** | 92% |
| **关键风险** | 深度自定义需ProseMirror知识 |
| **下一步** | 搭建PoC验证自定义节点、模板渲染、离线场景 |

## 五、Service Worker离线方案评估

### 5.1 Workbox框架评估

| 评估维度 | 评估结论 |
|----------|----------|
| **成熟度** | ★★★★★ Google官方维护，生产级验证 |
| **功能完整度** | 预缓存、运行时缓存、后台同步、离线回退全覆盖 |
| **缓存策略** | 提供6种内置策略（CacheFirst、NetworkFirst、StaleWhileRevalidate等） |
| **工具链** | workbox-cli、webpack/vite插件、构建时自动注入 |
| **社区** | GitHub 26k+ stars，npm周下载300万+ |

### 5.2 PWA离线安装支持

| 评估维度 | 评估结论 |
|----------|----------|
| **Chrome/Edge** | ✅ 完美支持，可安装到桌面 |
| **Firefox** | ⚠️ 已移除桌面PWA安装，但Service Worker正常工作 |
| **Safari (macOS)** | ⚠️ 有限支持，安装体验不如Chrome |
| **Safari (iOS)** | ⚠️ 支持添加到主屏幕，但有限制（无推送、7天不使用清除SW） |
| **Android** | ✅ Chrome/Edge完美支持 |

### 5.3 课伴离线方案设计

```
┌─────────────────────────────────────┐
│          Service Worker 层           │
├─────────────┬───────────────────────┤
│  静态资源缓存  │  动态数据缓存          │
│  (CacheFirst) │  (NetworkFirst)      │
├─────────────┼───────────────────────┤
│  JS/CSS/字体  │  API响应缓存          │
│  图片/模板    │  AI结果缓存           │
└─────────────┴───────────────────────┘
         ↕                  ↕
┌─────────────────────────────────────┐
│         IndexedDB (Dexie.js)        │
│        本地数据持久化存储              │
└─────────────────────────────────────┘
```

### 5.4 Safari兼容性风险

| 问题 | 影响 | 缓解措施 |
|------|------|----------|
| iOS Safari 7天不使用清除SW | 离线缓存可能丢失 | 提示用户定期访问，本地数据不受SW清除影响 |
| Safari不支持Web Push | 无法发送复习提醒通知 | 降级为应用内提醒 |
| Safari Private Mode下IndexedDB受限 | 无法存储数据 | 检测并提示用户退出隐私模式 |
| Safari对Cache API容量限制更严格 | 缓存空间不足 | 优化缓存策略，减少静态资源体积 |

### 5.5 评估结论

| 评估项 | 结论 |
|--------|------|
| **推荐方案** | Workbox + 自定义缓存策略 |
| **可行性评级** | **高** ✅ |
| **信心度** | 88% |
| **关键风险** | Safari/iOS兼容性 |
| **下一步** | 在Chrome/Firefox/Safari三端完成离线场景PoC |

## 六、综合评估结论

### 6.1 总体评级汇总

| 技术方向 | 可行性 | 信心度 | 主要风险 | 阻塞性 |
|----------|--------|--------|----------|--------|
| 本地存储（IndexedDB + Dexie.js） | **高** ✅ | 90% | Safari兼容性 | 否 |
| SM-2间隔重复算法 | **高** ✅ | 98% | 无显著风险 | 否 |
| AI可选接入 | **中高** ⚠️ | 75% | 网络依赖、成本控制 | 否（插件化隔离） |
| 富文本编辑器（TipTap） | **高** ✅ | 92% | 深度定制学习成本 | 否 |
| Service Worker离线（Workbox） | **高** ✅ | 88% | Safari兼容性 | 否 |

### 6.2 综合结论

> **项目在技术层面完全可行，核心技术栈成熟度高。**

关键结论如下：

1. **核心技术全部可落地**：本地存储、间隔重复算法、富文本编辑器、离线方案均有成熟的生产级实现，技术风险可控
2. **AI能力为增强项，非阻塞项**：AI采用插件化架构，不影响核心功能的开发和发布，可独立演进
3. **Safari兼容性是共性风险**：多个技术方向都存在Safari兼容问题，需统一制定Safari降级策略
4. **PoC验证优先**：建议在正式开发前，对本地存储大数据量场景、AI接口对接、离线全链路三个方向进行PoC验证
